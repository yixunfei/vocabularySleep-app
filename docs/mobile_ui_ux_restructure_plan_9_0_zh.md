# Vocabulary Sleep Flutter UI/UX 重构方案 9.0

更新时间：2026-03-11

## 1. 版本定位

9.0 是基于“第八版已部分修改”的最新复评版本，重点从“功能是否已接入”升级到“真实使用是否稳定顺手”。第八版已经在词库页做了多项有效改进，包括分批加载、索引弹层、滚动后回顶锚点、底部 tab 重复点击回顶等，这些都说明项目在快速迭代并且方向正确。  
但从当前代码状态看，产品仍处在“能力已出现、系统未完全成型”的阶段：用户能感知到改进，但在高频真实场景下，仍会遇到效率不稳定、路径不连续、性能链路不够稳的情况。

因此，9.0 的核心目标不是继续堆新入口，而是把第八版已经出现的能力收束为可长期使用的产品级体系：导航体系成型、状态切换可预期、长列表行为稳定、超大词库下仍可顺畅操作。

## 2. 当前版本复评结论（基于最新代码）

### 2.1 已经完成且有效的改进

1. 词库页已接入分批展示策略，`_visibleItemCount` 默认页大小为 20，并在滚动接近底部后自动增长。
2. 新增滚动监听与回顶悬浮锚点，`offset > 320` 后显示 `FloatingActionButton.small`，回顶路径比早期版本清晰。
3. 新增“索引工具”卡片与“展开索引”底部弹层，字母跳转不再只依赖窄横向条。
4. 底部导航重复点击当前词库 tab 已可触发回顶，移动端常见心智模型开始成立。
5. 播放页已补齐暂停态标识、上一词/下一词按钮与单词区域左右滑切换，操作连续性提升。

这些改动说明第八版不是局部修饰，而是已经触及到核心交互链路，值得保留并继续深化。

### 2.2 当前仍然影响“实际体验”的关键问题

#### A. 分批加载已做，但“全量计算链路”仍在

词库渲染虽由“全量展示”变为“分批展示”，但状态层仍是 `visibleWords` getter 即时过滤，每次读取都可能触发整表过滤。  
同时，词库页 build 阶段还会基于 `words` 全量做可用首字母统计，跳转链路也依赖 `visibleWords` 的即时结果。  
这意味着：展示层降压了，但计算层仍偏全量，在 1 万词以上词库和高频搜索输入下，仍可能出现卡顿或掉帧。

#### B. 当前列表仍是 `ListView(children)` 方式拼接

虽然 `displayedWords` 控制了数量，但当前仍通过 `for` 构建 children 列表，而不是 `ListView.builder` / `SliverList`。  
当加载条目不断增大时，每次 setState 都会重建已加载区，性能曲线仍不够理想。  
对于真实长时间使用，这会表现为“前期流畅，滚久后逐渐变重”。

#### C. 回顶逻辑与分页状态耦合，可能造成上下文丢失

当前 `_scrollToTop()` 不仅滚动回顶，还会把可见条目数量重置到第一页大小。  
这在性能上有好处，但在体验上会带来上下文断层：用户从深处回顶后再向下浏览，需要重复触发多次加载，容易产生“内容被收回去了”的感知。  
这个行为是否保留，应在 9.0 明确定义为产品策略，而不是默认技术副作用。

#### D. 索引工具仍是“首屏卡片”，不是“滚动中工具层”

第八版的索引可展开已经比旧版好，但索引工具仍位于内容流顶部。用户深滚后想再次跳转，仍需先回顶或依赖底部 tab 回顶。  
对于高频检索型产品，索引工具更适合逐步演进为可吸附工具层（至少保留一个简化入口），否则“已可展开”仍会被“触达成本高”抵消。

#### E. 悬浮动作层开始拥挤，未来扩展存在冲突风险

当前右下区域已同时承载“加词 FAB”和“回顶 FAB（条件出现）”。如果后续继续增加“跳到当前词”或“索引快捷动作”，右下角将进入冲突区。  
9.0 需要提前定义浮层优先级与显隐规则，否则每版都加一个按钮会快速退化为视觉噪声区。

#### F. 搜索体验仍缺少输入防抖与缓存策略

`setSearchQuery` 仍是每次输入直接更新状态并触发过滤链路。  
在大词库下，用户连续输入时会出现“每个字符都做一次完整计算”的压力。  
这会影响“搜索时感觉跟手”的核心体验，尤其在中低端机型更明显。

#### G. 播放页“对象直达”仍不够彻底

第八版已经加入上一词/下一词和滑动切词，值得肯定。  
但“点击当前词主体直接进入跟读/对象动作层”仍然不是主路径，当前依赖按钮组合与提示文案。  
对真实学习流来说，这仍然不够直觉，9.0 需要把当前词真正做成第一交互对象。

#### H. 测试覆盖与行为验收尚未跟上 UI 复杂度

现有测试主要集中在 `AppState` 逻辑，词库页新增的分页、锚点显示、索引展开、回顶联动等缺少 widget/integration 验证。  
这会导致“局部修好了，但跨场景回归不稳定”的风险升高。

## 3. 9.0 目标

9.0 建议以“稳定、高频、可预期”为目标，形成四个达成标准：

1. 超大词库（1 万+）下，滚动和搜索都保持可用流畅，不出现明显冻结。
2. 导航工具在用户真正需要时始终触达，不依赖回忆路径。
3. 回顶、跳转、分页三者行为一致，不制造内容上下文断层。
4. 播放与词库在“单词对象优先”上形成统一体验。

## 4. 方案 9.0 详细设计

### 4.1 词库页：从“分批展示”升级为“分层架构”

#### 4.1.1 渲染层升级（P0）

1. 将词条区域从 `ListView(children)` 迁移到 `ListView.builder` 或 `CustomScrollView + SliverList`。
2. 保留分批窗口思想，但改为“窗口增量 + 构建惰性”，避免每次 setState 重建整段 children。
3. 将“正在加载更多”的提示与加载触发解耦，触发可基于可视区域观察，而不是仅 `extentAfter` 阈值。

#### 4.1.2 计算层升级（P0）

1. 为 `visibleWords` 建立缓存模型（按词本版本、搜索词、搜索模式作为 key）。
2. 搜索输入加入轻量防抖（建议 120ms-200ms），避免每个字符都同步重算。
3. 可用首字母集合改为基于缓存结果派生，不在每次 build 时全量扫描。
4. 对模糊匹配场景逐步引入预处理索引，降低实时归一化成本。

#### 4.1.3 状态一致性策略（P0）

1. 明确定义回顶是否重置分页窗口。  
建议策略：默认不重置，只滚动回顶；仅在“内存回收条件”满足时做软重置，并给用户无感处理。
2. 跳转到目标词时，先确保窗口覆盖目标，再滚动定位，避免“跳转成功但列表未展开到位”的不确定性。
3. 搜索词变化后，分页窗口按结果规模自适应，而不是机械回到固定条目数。

### 4.2 导航体系：从“首屏工具区”升级为“滚动中导航系统”

#### 4.2.1 索引工具分层（P0）

1. 保留“展开索引”弹层作为完整入口。
2. 页面内仅保留简化预览，不再承担完整索引职责。
3. 将“浏览型索引（字母）”“定位型索引（前缀）”与“重置动作（回顶）”在视觉和语义上分层。

#### 4.2.2 工具触达升级（P1）

1. 逐步把索引入口演进为可吸附头部（`SliverPersistentHeader` 方向）。
2. 至少保证深滚状态下仍有一个低成本入口可再次打开索引弹层。
3. 避免用户每次都依赖“先回顶再跳转”的长路径。

#### 4.2.3 上下文自适应（P0）

1. 当结果集很小（如 <= 10）时，弱化或隐藏密集索引预览。
2. 当搜索激活且结果不多时，优先展示结果状态与清除动作。
3. 当结果集很大且非搜索态时，再强调字母索引预览。

### 4.3 浮层动作：统一规则防止右下角退化

#### 4.3.1 浮层优先级（P1）

1. `加词` 保持主 FAB。
2. `回顶` 保持次级 FAB，基于滚动阈值显示。
3. 若新增其他快捷动作，优先进入二级动作面板，不再直接叠第三个 FAB。

#### 4.3.2 显隐规则（P1）

1. 键盘弹起时隐藏次级 FAB。
2. 短列表或空结果时隐藏回顶 FAB。
3. 动态避让 mini player、导航栏和安全区，避免遮挡。

### 4.4 播放页：把“当前词”真正做成主对象

#### 4.4.1 对象直达（P1）

1. 在当前词头部或主内容区提供“点击即进入跟读”的主路径（按钮保留为次路径）。
2. 保持滑动切词与上下词按钮，但避免与主对象点击手势冲突。
3. 将“当前词正在播放”状态强化为对象状态，而非仅页面状态。

#### 4.4.2 大词库进度控件优化（P2）

1. 长列表下弱化离散 slider 的精确逐词定位角色。
2. 增加区段跳转或步进跳转，降低超长分段 slider 的操作成本。

### 4.5 验证与质量保障（P0/P1）

#### 4.5.1 新增测试建议

1. `LibraryPage` widget test：分页窗口增长、回顶锚点显隐、索引弹层打开与跳转。
2. 搜索场景 test：输入连续变化时结果稳定性与性能边界行为。
3. 导航联动 test：底部 tab 重复点击回顶与页面滚动状态同步。

#### 4.5.2 新增观测指标

1. 首屏可交互时间（词库页）。
2. 搜索输入到结果稳定的中位延迟。
3. 深滚场景下平均帧率与掉帧比。
4. 跳转成功率（触发后可见定位成功）。

## 5. 分阶段落地建议

### 阶段 A（优先级最高）

1. 渲染层迁移到 builder/sliver 惰性构建。
2. `visibleWords` 缓存化 + 搜索防抖。
3. 回顶/分页一致性策略确定并落地。

### 阶段 B

1. 索引入口分层与上下文自适应规则落地。
2. 右下浮层规则统一。
3. 索引触达从首屏工具区向“滚动中入口”演进。

### 阶段 C

1. 播放页对象直达完善。
2. 大词库进度控件策略优化。
3. 补齐 widget/integration 验收用例。

## 6. 9.0 验收标准（实操口径）

1. 在 1 万词词库下，词库页可快速进入并保持稳定滚动，不出现明显卡死。
2. 搜索输入连续键入时，界面无明显阻塞，结果更新不抖动。
3. 深滚状态下，用户可在 1 步内触达回顶与索引入口。
4. 回顶后用户不会感知“内容突然被回收导致再次找不到位置”的断层。
5. 索引弹层跳转后目标词可稳定进入可见区，不出现“跳了但没到”。
6. 播放页可直接围绕当前词进入跟读，不必先定位二级按钮。
7. 新增行为具备自动化测试覆盖，回归时可快速发现破坏性变更。

## 7. 最终建议

第八版已经把“可见问题”推进到了“可用能力”，这是非常好的阶段成果。  
第九版的任务，是把这些能力做成可长期信任的系统。  
建议 9.0 以“性能托底 + 导航成型”作为双主线优先推进，先确保高频场景稳，再做视觉和交互细节继续打磨。这样可以最大化每一轮修改对真实用户体验的正向增益。
