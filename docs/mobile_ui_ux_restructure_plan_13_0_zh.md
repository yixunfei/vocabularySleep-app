# Vocabulary Sleep Flutter UI/UX 反馈方案 13.0

更新时间：2026-03-12

## 1. 版本定位

这一次 13.0 的复评，不再沿用前几轮“先把明显错位、半截按钮、语言缺字补齐”的问题视角，而是把当前项目放到一个更成熟的阶段重新判断：

当前 `flutter_app` 已经不再是“移动端基础能力缺失”的状态，而是进入了“能力越来越多，但设计规则、响应式策略和多语言治理是否能同步跟上”的阶段。

换句话说，项目现在最大的风险，不是某一个页面还不够好看，而是：

- 应用级本地化能力已经接通，但 UI 文案治理仍然是双轨并行，后续页面越多，维护成本越高。
- 局部页面已经开始修复适配问题，但断点规则分散在各组件和页面里，尚未形成统一的移动端布局体系。
- 新增的专注页、数据管理页等能力让产品更完整了，但也把“真实使用体验”推到了新的质量门槛前。

因此，13.0 的核心结论是：

当前项目已经从“修页面问题”进入“补产品规则”的阶段。接下来如果继续只做局部修补，短期能看起来越来越完整，但长期会越来越依赖个别页面手工兜底，而不是依赖稳定的一套 UI/UX 设计系统。

- 页面适配是否已经从“局部修一下”走向“形成统一响应式策略”。
- 真实体验是否已经覆盖到高频使用场景，而不是停留在单页视觉修正。

## 3. 当前版本值得肯定的进展

先明确一点：当前版本并不是“改了很多但仍然没起色”。相反，这一轮代码已经出现了几项非常关键的正向进展，这些进展意味着项目已经具备继续做深的基础。

### 3.1 应用级 locale 链路已经补齐

`main.dart` 当前已经接入：

- `flutter_localizations`
- `locale`
- `supportedLocales`

`SettingsService` 也不再把默认语言固定为中文，而是引入了 `system` 模式；`AppState` 则通过 `WidgetsBindingObserver` 和 `didChangeLocales` 处理系统语言变化。

这说明项目已经从“只是业务文案切换语言”进入“应用运行环境真的能跟随系统语言”的阶段。这个基础能力非常重要，因为它决定了后面做多语言时，不会再卡在最底层。

### 3.2 中文词典级文案已经补齐

本轮重新统计 `AppI18n` 词典后，当前英文基准 key 总量为 `391`，中文也已经达到 `391`，说明中文主词典已与英文基准表齐平。

这和前几轮最大的区别是：

现在已经不能再把“中文翻译大量缺失”作为主结论。中文环境下的核心问题，已经从“缺字”转向“文案来源是否统一、是否所有页面都走统一词典”。

### 3.3 多语言入口和系统词本本地化比之前更成熟

当前项目已经具备以下正向能力：

- 语言设置页支持“跟随系统”和“手动指定语言”。
- `AppI18n.supportedLanguages` 已经接入中文、英文、日文、德文、法文、西文。
- `wordbook_localization.dart` 已为收藏、任务等系统词本提供多语言显示名，而不是继续直接显示数据库原始名称。

这代表团队已经不再把多语言理解为“简单切一个中文/英文按钮”，而是在逐步把系统层显示名也纳入产品语义体系。

### 3.4 页面适配开始出现真正的修复动作

和上一轮相比，多个页面已经不再完全忽略窄宽度问题：

- `SettingTile` 已经开始区分图标型 trailing 和文本型 trailing，并在窄宽度下允许堆叠。
- `SectionHeader` 已经对小宽度做紧凑处理。
- `PageHeader` 已支持窄宽度时把 action 下放到标题块下方。
- `AppearanceStudioPage` 已引入 `LayoutBuilder`、`compactWidth`、`isExpanded: true` 等修正。
- `LibraryPage` 的索引预览已经从容易截断的横向排布改成了 `Wrap`。

这说明当前项目的设计质量并不是停在原地，而是已经开始向“移动端体验感知”靠近了。

### 3.5 词库页的大数据体验已经开始从“整页硬扛”转向“分段加载”

`LibraryPage` 当前已经引入：

- `_pageSize = 20`
- `_visibleItemCount`
- 滚动触底后的增量加载
- 面向目标索引的估算滚动与逐步对齐
- 回到顶部与字母索引的浮层入口

这意味着之前“大词库一次性塞满导致滚动卡顿”的风险，已经开始被结构性处理，而不是只靠优化单个 cell。

### 3.6 数据重置能力比之前更像一个正式产品功能

`DataManagementPage` 现在已经提供：

- 删除用户数据
- 删除前确认
- 自动创建安全备份
- 最近一次安全备份路径展示

`AppState.resetUserData()` 与 `DatabaseService.resetUserData()` 也已经形成闭环。

这说明“重置数据”已经不只是一个技术动作，而是开始具备面向用户的产品化表达。

## 4. 当前最需要重视的问题

## 4.1 多语言本地化：底层能力已成立，但治理方式仍然不够成熟

### 4.1.1 中文已经补齐，但非中文覆盖依然明显不足

本轮重新统计后，当前翻译覆盖情况大致如下：

- 英文基准 key：`391`
- 中文：`391 / 391`
- 日文：`307 / 391`，仍缺 `84`
- 德文：`307 / 391`，仍缺 `84`
- 法文：`307 / 391`，仍缺 `84`
- 西文：`307 / 391`，仍缺 `84`

这说明当前项目的多语言问题，已经不是“有没有国际化能力”，而是“是否真的把国际化当成产品长期能力治理”。

如果继续维持现状，会出现一个典型结果：

中文用户会觉得“现在差不多能用了”，但任何切到日语、德语、法语、西语的用户，仍然会感知到大量英文回退，这会直接削弱产品的完成度印象。

### 4.1.2 当前 UI 文案仍是双轨体系，后续维护会越来越重

当前项目存在两套并行的文案获取方式：

- 一套是 `AppI18n.t('key')` 的词典式方案。
- 一套是 `pickUiText(i18n, zh: ..., en: ...)` 的页面内联方案。

本轮统计到 `pickUiText` 在代码中共出现 `445` 次，其中 `1` 次是函数定义本身；剩余 `444` 处真实调用里：

- 只有 `4` 处同时补了 `zh/en/ja/de/fr/es`
- `1` 处为部分补充
- `439` 处仍然只是 `zh/en` 双语分支

这就是当前多语言治理真正的核心风险。

风险不在于现在不能显示，而在于后续每新增一个页面，都可能重复出现下面的问题：

- 页面作者直接写 `zh/en`，不会第一时间补完整多语言。
- 新增能力先写在 UI 层，没进入统一词典。
- 翻译缺口变成“分散在各页面里的隐性债务”，难以统一审计。

从产品设计角度看，这会直接带来三个体验后果：

- 语言一致性差，同一层级页面会混用统一词典文案和临时页面文案。
- 翻译质量难维护，后续校对没有单一收口点。
- 新语言扩展成本越来越高，因为 UI 层本身携带了大量文案分支。

### 4.1.3 当前最值得推动的，不是继续补单页翻译，而是收拢翻译治理方式

当前阶段的重点不应该是“再把几个英文标题翻一翻”，而应该是：

- 把高频页面文案从 `pickUiText` 回收进 `AppI18n` 主词典。
- 为页面临时文案建立“新增文案必须先进词典”的约束。
- 给翻译覆盖率提供自动化校验，避免每次靠人工肉眼排查。

这会比继续手动补零星字符串更有价值，因为它是在解决翻译系统本身的可持续性问题。

## 4.2 页面适配：修复已经开始，但响应式策略仍然是零散的

### 4.2.1 当前不是没有适配，而是适配规则没有收口成体系

目前可以看到多个组件已经开始自带窄宽度判断，但阈值明显分散：

- `SettingTile`：`340`
- `SectionHeader`：`360`
- `LibraryPage` 某些索引策略：`360 / 380`
- `PageHeader`：`390`
- `AppearanceStudioPage`：`390`

这会带来一个典型问题：

同一台设备上，页面 A 进入紧凑态了，页面 B 还在标准态，页面 C 又进入了另一个阈值下的半紧凑态。用户不会知道背后是多个组件各写了一套断点，他只会感知为“这个应用的版式行为不稳定”。

因此，当前适配问题的本质，已经不是某个控件没写 `LayoutBuilder`，而是：

项目还没有一套统一的移动端宽度等级和组件响应规则。

### 4.2.2 `SettingTile` 和头部组件已改善，但仍停留在个别修补

`SettingTile` 当前已经比之前明显更好，不再总是把箭头挤到下一行左下角，这是一个很关键的修复。

但从系统角度看，它仍然存在两个问题：

- 组件是否堆叠，仍然依赖内部条件判断，而不是统一的页面宽度等级。
- 当前策略主要解决“不要错位”，还没有进一步解决“在大字体、多语言、长副标题场景下是否仍稳定”。

`PageHeader` 和 `SectionHeader` 也有类似情况。它们已经能处理窄宽度，但每个组件都在自己决定何时换行、何时下沉 action。这种局部判断短期有效，长期会让整个系统越来越难统一。

### 4.2.3 `AppearanceStudioPage` 已进入“可用修复”，但还没进入“可持续设计”

外观工作室这一页这次比之前成熟很多：

- 已经有 `LayoutBuilder`
- 已经有 `compactWidth`
- 下拉框已加 `isExpanded: true`
- slider 与 section padding 也开始有紧凑态

这说明前面“中文环境下大量控件挤压、下拉宽度不稳”的问题已经得到明显改善。

但它依然存在一个结构性问题：

这是一个内容密度很高的长单页表单，而当前的“紧凑化”主要还是通过减少间距、缩小 padding、让控件更愿意收缩来实现。这样的方式可以缓解局部拥挤，但不能真正解决“信息层级过重、连续操作成本高、长时间滚动疲劳”的问题。

从设计上看，外观工作室下一阶段更适合做的是：

- 从单纯调 spacing，升级为“分组卡片 + 紧凑组件模板”的统一方案。
- 把高频项、预设项、专家项做层级拆分，而不是继续把所有配置堆在一个长列表里。
- 把“窄宽度策略”从页面局部变量沉淀成通用表单规范。

### 4.2.4 `FocusPage` 是当前页面适配中最值得优先处理的新短板

专注页是这次最值得单独强调的地方。它不是“很差”，而是“功能已经成立，但移动端结构明显还不够成熟”。

从代码上看，这一页目前存在几个明显的窄宽度风险点：

- 番茄钟配置区域使用一个 `Row` 放置三列 `Expanded` 字段，中间固定 `16` 间距。
- 运行中控制区把继续、跳过、停止三个按钮放在同一行。
- 今日统计仍是横向并排两组信息。
- 快速笔记区域使用固定高度横向卡片列表，卡片宽度固定为 `160`。

这些结构在中等宽度设备上大概率还能工作，但在以下场景下会迅速暴露问题：

- 小屏 Android 设备
- 大字体模式
- 非中文语言下文案变长
- 横屏或分屏状态

更重要的是，这一页现在已经不是边缘功能，而是产品能力扩展后的新入口。如果这一页体验不稳，会让用户觉得产品在“新增能力”上仍然是功能优先、体验滞后。

## 4.3 真实体验：当前已经从“页面美观”进入“流程是否顺手”的阶段

### 4.3.1 词库页的大数据体验明显改善，但交互入口仍偏多

词库页当前有几个值得肯定的进展：

- 已开始采用增量展示，缓解大词库一次性构建的压力。
- 索引预览改成 `Wrap`，之前那种“半截字母 chip”式的观感问题已经明显缓解。
- 回到顶部、字母索引等快捷入口已经具备。

但从体验角度看，当前词库页的入口职责仍然略显分散：

- 搜索框
- 搜索模式分段控件
- 索引工具卡片
- 字母预览
- 前缀跳转
- 浮层字母索引按钮
- 浮层回顶按钮
- 加词按钮

这些入口单独看都合理，但组合在一起时，用户容易感觉页面“什么都给了，但主次不够清楚”。

当前更适合做的，不是继续增加入口，而是进一步收口职责：

- 顶部工具卡片负责“发现和跳转”。
- 浮层只保留最关键的一个辅助动作。
- 分段搜索在窄宽度下避免继续依赖横向滚动作为主要解法。

现在词库页已经不缺功能按钮了，缺的是更清晰的操作主线。

### 4.3.2 专注页的最大问题不是单页排版，而是和“学词场景”连接还不够深

当前 `FocusPage` 已经具备番茄钟、Todo、快速笔记，这本身是一次很大的能力拓展。

但站在产品体验上，这一页目前仍偏向“通用专注工具页”，还没有完全长成“为词汇学习服务的专注页”。例如：

- 当前页面主要围绕时间和任务组织，没有明显强化“专注过程中继续和单词互动”的路径。
- 和词库、播放、跟读之间的上下文衔接仍然偏弱。
- 从用户感知上，专注页更像是被添加进来的一个新模块，而不是深度融入学习主流程的一环。

这并不代表现在要立刻重构成功能整合大页，而是说明下一阶段在 UX 上要回答一个问题：

专注页究竟只是一个辅助工具，还是要成为“学习中场景切换”的核心容器？

如果答案是后者，那么这页就需要在后续版本里加强和词条播放、跟读练习、学习目标之间的连接。

### 4.3.3 数据管理已经有“重置”能力，但“恢复”仍然是体验闭环缺口

当前 `DataManagementPage` 已经在“删之前先备份”这件事上做得比很多产品都更谨慎，这一点值得肯定。

但从用户信心角度看，这一页还缺最后一步：

当前恢复入口仍然只是占位说明，而不是可实际使用的恢复流程。

这会造成一个心理落差：

- 用户知道系统会自动创建安全备份。
- 但用户还不能通过产品界面真的浏览备份、确认来源、执行恢复。

也就是说，“安全备份”目前更像后台技术保障，而不是已经让用户真正可感知、可操作的恢复能力。

从 UX 来看，只要恢复链路没有落地，重置功能就始终会带一点心理负担。

## 5. 方案 13.0：建议的重构方向

## 5.1 P0：优先处理，直接影响产品成熟度

### 5.1.1 建立统一的本地化治理规则

建议立即把多语言治理从“页面自带文案”切换到“词典优先”模式，核心动作包括：

- 规定新增页面文案优先进入 `AppI18n`，减少继续扩张 `pickUiText`。
- 为 `pickUiText` 设定明确使用边界，例如只允许少量临时拼装文案或极个别展示型文案。
- 增加本地化覆盖率脚本，至少校验英文基准与其他语言缺失 key。
- 在提测前加入“中文、英文、任意第三语言”的三语言 smoke 检查。

这一项的价值非常高，因为它是在避免项目越做越难翻，而不是只补当前几页。

### 5.1.2 把断点从“页面自己写”升级为“系统统一定义”

建议在 UI 层抽象统一的宽度等级，例如：

- 紧凑宽度
- 标准手机宽度
- 宽手机 / 小平板宽度

然后让 `PageHeader`、`SectionHeader`、`SettingTile`、表单区块、浮层按钮等都基于同一套宽度等级做响应，而不是继续各自维护 `340/360/380/390`。

这一步完成后，页面适配问题会出现一个根本性变化：

后续不再是“哪里挤了修哪里”，而是“页面自动继承统一的窄宽度策略”。

### 5.1.3 优先修复 `FocusPage` 的移动端结构

建议把专注页列为当前移动端适配的第一优先级页面，具体方向包括：

- 番茄钟配置区从三列并排改为可折行或双列/单列自适应布局。
- 运行控制按钮从固定横排改为 `Wrap` 或分层主次按钮。
- 今日统计改成卡片栅格或上下堆叠，避免大字体时横向挤压。
- 快速笔记卡片改为可随宽度变化的自适应尺寸，而不是长期固定 `160`。

专注页是当前最像“新增能力已经到位，但设计规则还没补齐”的代表页面，先把它做稳，能明显提升整个产品的完成度。

### 5.1.4 让“恢复备份”从占位变成可落地入口

建议把数据管理中的恢复流程至少做到最小可用版本：

- 可查看最近备份列表
- 可显示备份时间与来源
- 可执行一次明确确认的恢复
- 可在恢复前再次提示当前数据风险

这会让“删除用户数据”从技术安全变成真正的用户安全。

## 5.2 P1：第二阶段处理，提升整体一致性和体验顺滑度

### 5.2.1 收口词库页交互层级

建议对词库页做一次“入口职责收口”，重点不是删功能，而是理清哪些属于主流程、哪些属于辅助流程：

- 保留一个稳定的主跳转入口。
- 减少窄宽度下需要横向滚动才能完整使用的关键控件。
- 重新梳理浮层按钮与顶部工具卡片的分工。

词库页现在的主要问题不是缺能力，而是入口已经开始拥挤，需要更清晰的层级和优先级。

### 5.2.2 为表单型页面沉淀通用紧凑模板

建议把外观工作室、语言设置、语音设置、识别设置这类页面统一纳入同一套表单模板，包括：

- 标题块规范
- 分组卡片规范
- 下拉/滑杆/开关的紧凑态规范
- 大字体和多语言下的留白与换行规则

这样后续新增设置页时，就不需要每个页面再各写一套 `compactWidth` 和 spacing 逻辑。

### 5.2.3 补上非中文语言的完整覆盖

当前中文已经达标，下一阶段应把重点转向：

- 日文、德文、法文、西文的缺失 key 补齐
- 系统页和设置页的高频文案优先补齐
- 建立术语统一表，避免同一概念在不同页出现不同翻译

如果要做国际化产品，现在已经到了必须从“支持多语言”升级到“经营多语言”的阶段。

## 5.3 P2：第三阶段处理，提升产品感知与可持续质量

### 5.3.1 建立更完整的真实设备矩阵验证

建议至少固定几组设计验收场景：

- 320 宽小屏
- 360 宽主流 Android
- 390 宽 iPhone 级
- 大字体
- 非中文长文案语言

只有把这些场景固定成验收标准，页面适配才不会每次都靠人工感觉。

### 5.3.2 给本地化和 UI 回归补自动化门槛

当前已有部分 `ui_smoke_test.dart`，这是一个不错的开始，但还不够覆盖新风险。建议继续增加：

- `FocusPage` 的窄宽度 smoke test
- 多语言页面快照或关键文本断言
- 大词库分页加载和索引跳转回归
- 数据重置与恢复流程的回归测试

自动化的意义不是替代设计判断，而是防止“旧问题修好了，新页面又把同类问题带回来”。

## 6. 建议的执行节奏

如果以实际落地为目标，建议按下面的节奏推进：

### 第一阶段：1 周内

- 完成本地化治理规则定稿
- 抽出统一宽度等级与基础响应式工具
- 优先修正 `FocusPage` 的窄宽度结构

### 第二阶段：1 到 2 周

- 把高频页面文案逐步回收到 `AppI18n`
- 为设置类页面沉淀统一紧凑模板
- 补齐非中文语言缺失 key

### 第三阶段：持续优化

- 重构词库页入口层级
- 做恢复备份最小可用流
- 补齐自动化回归矩阵

## 7. 建议验收标准

为了避免方案落地后仍停留在“主观感觉变好一些”，建议把以下内容作为下一轮验收标准：

- 中文、英文、日文三个语言环境下，设置类与管理类页面不出现明显英文回退。
- 所有核心基础组件统一使用同一套宽度等级，不再各自维护零散阈值。
- `FocusPage` 在 320 到 390 宽度下无横向拥挤、无主要按钮碰撞、无统计块压缩失衡。
- `AppearanceStudioPage` 在紧凑态下不出现表单控件宽度异常和长文本拥挤。
- `LibraryPage` 在大词库场景下滚动流畅，索引跳转与回顶入口职责清晰。
- 数据管理页至少具备可感知、可验证的备份恢复链路。

## 8. 最终结论

当前项目的状态，可以概括为一句话：

基础能力已经明显成熟，但系统规则还没有完全成熟。

本轮最值得肯定的是：

- 应用级语言体系已经接通
- 中文词典已补齐
- 页面适配开始出现真正修复
- 词库页大数据体验已经从结构上改善
- 数据重置功能开始具备产品化表达

本轮最需要继续推进的是：

- 多语言从“可显示”升级到“可治理”
- 页面适配从“局部修补”升级到“统一响应式系统”
- 专注页从“功能成立”升级到“移动端体验成熟”
- 数据管理从“能备份”升级到“能恢复”

如果 13.0 能把这四件事收口，当前 `flutter_app` 的完成度会出现一次非常明显的跃迁：它不只是“Flutter 重构版更像移动端了”，而会开始真正具备一个移动产品应有的设计一致性、体验稳定性和多语言扩展能力。
