# Toolbox UI 设计规范基线

## 文档版本

| 版本 | 日期 | 状态 | 作者 |

---

## 1. 目的

本规范用于约束 `toolbox` 模块后续 UI 精修时的统一设计语言，确保：

- 视觉层持续收口，不再出现页面之间风格漂移
- 新增或重构的组件能与现有页面自然拼接
- “只动 UI、不动逻辑”的迭代有明确边界，降低误改业务行为的风险
- 页面在移动端一屏信息组织、触控体验和氛围表达上保持稳定一致

本规范是对以下文档的执行性补充：

- `docs/toolbox_design/TOOLBOX_DESIGN_REVIEW.md`
- `docs/toolbox_design/TOOLBOX_ANIMATION_SPEC.md`

---

## 2. 适用范围

适用于以下 toolbox 页面及其后续同类页面：

- 工具箱入口页
- 疗愈音钵
- 呼吸训练
- 舒缓音乐
- 静心念珠
- 禅意沙盘
- 后续任何沿用 `toolbox` 展示语言的新页面

不适用于：

- 业务逻辑、状态管理、持久化结构调整
- 音频播放逻辑、计时逻辑、数据模型和导航语义变更
- 与 `toolbox` 视觉语言无关的全局主题重构

---

## 3. 设计总原则

### 3.1 移动端优先

- 关键主舞台信息尽量在手机首屏内完成识别
- 优先避免水平滚动；若必须横向布局，需保证可见边界、触控冗余和折行策略
- 在 375dp 宽度下，标题、说明、主控件和状态反馈不得互相遮挡
- 单手操作优先，频繁操作控件应尽量落在自然拇指热区

### 3.2 先收层次，再加装饰

- 优先解决信息分组、主次节奏、留白和对齐，再考虑渐变、辉光和纹理
- 任何装饰性元素不得抢夺主操作按钮和主状态信息的注意力
- 氛围层应该服务沉浸感，不应制造机械感、噪声感和视觉疲劳

### 3.3 柔和但不松散

- toolbox 的整体气质应是克制、舒缓、细腻，而不是寡淡或虚弱
- 面板、按钮和信息 pill 应保持柔和圆角与浅层体积感，但不能缺少边界
- 大面积留白要有组织感，避免出现“像占位稿”的空洞页面

### 3.4 统一语法，保留模块个性

- 页面骨架、卡片语言、间距和交互反馈统一
- 各工具模块通过 accent 色、氛围背景、局部图形节奏体现差异
- 不允许每个页面自行发明一套新的卡片边框、按钮风格或信息标签样式

---

## 4. 页面骨架规范

### 4.1 统一结构

toolbox 页面默认按以下顺序组织：

1. 顶部导航区
2. 主舞台信息区
3. 当前状态或模式切换区
4. 主操作区
5. 辅助设置区
6. 次级说明或安全反馈区

### 4.2 主舞台信息区

主舞台信息区是每个页面最优先精修的部位，应满足：

- 在首屏即可识别“这是哪个工具、当前是什么状态、下一步可以做什么”
- 至少包含一个主标题、一个模式或上下文标签、一个当前状态摘要
- 若存在错误、加载、同步中状态，必须以独立反馈块呈现，不要混入正文
- 若舞台已有强氛围背景，信息容器必须额外提供半透明面板或边框承托

主舞台信息区禁止：

- 直接把说明文字散落在背景层上
- 把模式、标题、状态和加载信息全部做成同一层级的文本
- 为了“空灵感”牺牲可读性，导致亮底浅字或暗底低对比

### 4.3 内容宽度

- 页面最大内容宽度保持在 600dp
- 面向手机首屏的核心信息推荐控制在 320dp 到 560dp 的阅读宽度内
- 宽屏下允许居中，但不允许让信息跨行节奏明显变坏

---

## 5. 卡片与面板规范

### 5.1 基础卡片语言

toolbox 统一使用“柔和面板”作为基础承载层，特点如下：

- 有边界：需要细边框或透明描边
- 有体积：允许轻量阴影或渐变
- 有温度：颜色不要纯白纯灰硬切
- 有克制：阴影和辉光都不应过厚

### 5.2 推荐参数

- 常规卡片圆角：16dp 到 24dp
- 胶囊类控件圆角：999dp
- 面板内边距：12dp 到 20dp
- 卡片间距：12dp 到 16dp
- 弱阴影模糊：10 到 18
- 强调阴影模糊：18 到 28，仅用于主舞台或当前选中态

### 5.3 分层规则

- 主舞台卡片可以使用双层色面或轻微渐变
- 次级设置卡仅保留边界与低强度阴影
- 危险或异常状态卡使用语义色背景，但必须降低饱和度，避免刺眼

禁止：

- 同一页面中混用多种不相关的卡片语言
- 用过粗边框模拟层次
- 用过深纯黑阴影制造“悬浮感”

---

## 6. 信息组织规范

### 6.1 文本层级

每个工具页至少应有以下三个文字层级：

- 主标题：当前工具或当前主状态
- 次级标签：模式、轨道、阶段、频率、场景等上下文
- 说明文本：描述、指导、提示、补充反馈

推荐规则：

- 主标题权重最高，但字重和字号应服务稳定阅读，不追求夸张视觉
- 次级标签优先做成 pill、chip 或短标签，不要堆成长句
- 说明文本长度控制在 1 到 3 行，过长内容应下沉到独立说明区

### 6.2 状态反馈

状态反馈必须单独可识别：

- 加载中：独立加载条或进度块
- 错误：独立风险色容器
- 已选中：靠形状、边框、色块、阴影联合表达，不只靠颜色
- 不可用：降低透明度并保留语义解释

### 6.3 信息 pill

适合承载：

- 当前模式
- 当前轨道
- 当前阶段
- 当前材质
- 当前频率

不适合承载：

- 一整段说明文案
- 错误描述全文
- 多层语义混杂的复合信息

## 7. 色彩规范

### 7.1 色彩角色

toolbox 页面中的颜色应分为以下角色：

- `base`: 背景和基础面板
- `accent`: 当前工具主强调色
- `support`: 次强调色或伴随色
- `text_primary`: 主文字
- `text_secondary`: 次级文字
- `border`: 边界
- `danger / warning / success`: 语义反馈色

### 7.2 使用原则

- 一个页面只保留 1 个主强调色和 1 个辅强调色
- 氛围背景允许使用同色系渐变，但不要叠加超过两种高饱和主色
- 语义色只用于状态反馈，不应抢占品牌色或模块色
- 暗色氛围页面中，强调色要负责聚焦；浅色面板则负责可读性

### 7.3 一致性要求

- 同类信息在不同页面中不能今天是描边 pill、明天是纯底 chip、后天又变成图标标签
- 相同语义的禁用态、错误态、加载态要尽量复用相近表达
- 若新增页面需要新 accent，必须先确认其与既有模块不会互相混淆

---

## 8. 动效与氛围规范

### 8.1 动效原则

- 交互动效要短、准、轻
- 氛围动效要慢、稳、柔
- 页面中同一层级的动效时长和 easing 应统一

### 8.2 推荐节奏

- 轻触反馈：80ms 到 140ms
- 选中态过渡：180ms 到 280ms
- 卡片展开或内容切换：220ms 到 380ms
- 主舞台氛围循环：4s 到 8s

### 8.3 动效边界

允许：

- 轻微缩放
- 柔和透明度变化
- 小幅位移
- 低频率辉光脉动
- 不完全对称的背景漂移

避免：

- 高频闪烁
- 大幅旋转
- 永不停歇的强光呼吸
- 机械重复且节奏明显的往返运动
- 为了“高级感”引入影响阅读和性能的复杂噪声动画

---

## 9. 组件稳定性规范

### 9.1 共享优先

以下样式应优先收敛为共享组件或共享 token：

- 主舞台信息容器
- 柔和面板
- 胶囊型选择器
- 信息 pill
- 主操作图标按钮
- 动效时长与 easing

### 9.2 组件职责边界

- 组件只承载视觉组织和局部交互反馈
- 不在样式重构中顺手改动状态来源、事件语义或数据计算
- 样式组件命名应体现用途，而不是视觉噱头

推荐命名：

- `...panel.dart`
- `...pill.dart`
- `...chip.dart`
- `...header.dart`
- `...section.dart`

避免命名：

- `magic_widget.dart`
- `common_ui.dart`
- `beautiful_card.dart`

### 9.3 文件组织

- 页面文件负责结构编排，不要继续膨胀成超长样式仓库
- 复杂页面应拆出 header、panel、controls、tokens、copy 等子文件
- 展示层常量统一沉淀，不要在页面中散落十几个局部 `Color(...)`

---

## 10. 可维护性规范

### 10.1 只动 UI、不动逻辑

在该约束下允许修改：

- Widget 树结构调整
- 样式常量抽取
- 共享组件收敛
- 文案容器重排
- 动效参数整理
- 页面内局部命名和文件组织优化

在该约束下禁止修改：

- 业务状态机
- 事件触发逻辑
- 计时、播放、持久化逻辑
- 路由跳转语义
- 数据模型和服务层行为

### 10.2 精修顺序

后续视觉层问题优先按以下顺序处理：

1. 主舞台顶部信息区是否清晰
2. 主要交互控件是否可识别、可触达
3. 当前状态和反馈是否独立可读
4. 卡片与模块之间的层次是否统一
5. 背景氛围是否自然且不干扰内容
6. 局部装饰是否需要再增强

### 10.3 风险提示

遇到以下情况，应暂停直接改代码，先补设计判断：

- 视觉问题表面上来自布局，实则牵涉状态逻辑
- 需要新增“模式”、“阶段”或“轨道”的语义定义
- 某个页面为了满足设计反馈，必须改动操作流程
- 现有组件无法承载新设计，且新组件会影响多个页面语义

---

## 11. 视觉验收清单

每次 toolbox UI 精修完成后，至少检查以下项目：

- 首屏能否在 3 秒内理解页面用途
- 主标题、模式、当前状态是否分层明确
- 核心操作控件最小触控区域是否达到 48dp
- 卡片边界、阴影、圆角是否与已收口页面一致
- 错误、加载、选中态是否独立可识别
- 背景动效是否存在机械重复感
- 新增样式是否已经优先复用现有共享组件或 token
- 是否严格做到“只动 UI、不动逻辑”

---

## 12. 后续维护建议

- 新页面进入 toolbox 前，先从本规范中选定其主舞台结构、卡片语言和色彩角色
- 新一轮设计反馈优先标注“视觉层问题”还是“交互语义问题”，避免混修
- 若某种组件在两个以上页面重复出现，应尽快抽取共享层，而不是继续复制粘贴
- 若未来需要更完整的设计系统，可在本规范基础上继续拆分为：颜色令牌、组件规范、页面模板三份文档
