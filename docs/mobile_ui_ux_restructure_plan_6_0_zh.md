# Vocabulary Sleep Flutter UI/UX 重构方案 6.0

更新时间：2026-03-10

## 1. 版本说明

本文件是在 1.0 到 5.0 的连续评估基础上，结合当前第五版 UI 的最新代码状态，以及本轮你提供的两张截图问题，形成的 6.0 版本方案。与前几版不同，6.0 不是泛化地讨论“整体体验还可以怎么优化”，而是明确围绕截图中已经暴露出来的界面成品问题来重新判断当前项目。

从当前代码状态看，第五版 UI 确实已经解决了一批 5.0 阶段的关键问题。例如：

- 单词级跟读入口已经不再只存在于练习中心，而是已经打通到播放页词卡、词库列表、词条详情页、复习页等多个位置。
- 词库跳转已经从固定高度估算，升级为基于 `GlobalKey + Scrollable.ensureVisible` 的真实定位方式。
- `PageHeader`、`SettingTile`、`WordbookSwitcher`、`MiniPlayer` 等组件已经开始加入窄屏 fallback。
- `SegmentedButton` 的主题层已经补上了选中态样式，按钮层级问题也在开始治理。

这说明 5.0 里提出的“组件规范化”和“单词级操作路径”并不是停留在纸面上，而是已经真正往前推进了。

但与此同时，本轮截图也非常明确地说明：当前 UI 已经从“功能不完整”阶段，进入了“局部成品感不够、视觉秩序还不稳定”的阶段。现在最主要的问题，不再是某个入口缺失，而是同一个页面内部的排版逻辑、控件权重、信息块节奏和主次关系，仍然会在真实手机界面上暴露出明显的不自然感。

因此，6.0 的关键词应当是：

从“组件已接入响应式”走向“页面内部视觉秩序真正成型”。

## 2. 本次截图问题的核心判断

你提供的两张截图，实际上指向了两类非常典型、也非常关键的移动端 UI 问题。

第一类问题，是词卡内部的信息块布局不稳定。第一张图中你圈出的区域，本质上不是“字段没显示完整”，而是字段块的布局方式本身不够成熟。当前 `WordCard` 对额外字段采用的是 `Wrap + 不定宽 Container` 的方式，这会导致字段卡片宽度完全由内容决定。结果就是：有些字段块会占满整行，有些字段块只占很窄一段，剩下大片空白区域被留在右侧，看起来像“布局没对齐”、“卡片缺了一块”或者“界面没排完”。这不是纯视觉瑕疵，而是信息编排逻辑没有产品化的表现。

第二类问题，是主操作按钮和次操作按钮的视觉权重仍然不够稳定。第二张图里你圈出的空白按钮区域，虽然从代码结构上它应该是播放主按钮所在位置，但在实际视觉表现上，它几乎像一个没有内容的浅色胶囊。无论这是来自按钮前景色对比不足、主题色过浅、主按钮样式退化，还是局部状态下按钮内容没有建立足够强的可见性，它都指向同一个结论：主操作按钮还没有形成真正可靠的视觉优先级，用户无法一眼判断“这里是当前最重要的动作”。

因此，6.0 的核心问题已经非常清楚，不是继续做新功能，而是把页面里的信息块、字段卡、按钮、快捷操作和底部控制区真正排成一套有秩序的成品界面。

## 3. 相比 5.0，当前版本已经完成的关键进展

首先，5.0 中提出的“跟读必须成为单词级独立模式”已经明显推进。当前代码中，`FollowAlongPage` 已经可以从以下场景直接进入：

- 播放页的当前词卡
- 词库列表项
- 词条详情页
- 复习页条目
- 练习中心

这意味着跟读已经不再只是练习页里的一个附属按钮，而是真正开始成为“围绕单词对象的独立模式”。这一点在 6.0 里应被视为已完成的重要基础，而不是继续反复讨论的目标。

第二，词库页的“条目错位感”底层原因之一已经被修掉。此前跳转使用固定行高估算，容易导致滚动定位不准。当前已经换成基于 `KeyedSubtree` 和 `Scrollable.ensureVisible` 的真实定位，这说明“列表不稳”的问题已经从逻辑层面迈出了一大步。

第三，响应式已经开始进入组件层。`PageHeader`、`SettingTile`、`WordbookSwitcher`、`MiniPlayer` 都新增了基于宽度的分支逻辑，这说明 5.0 强调的“不要再靠默认控件硬扛窄屏”已经开始真正落地。

第四，主题层也已经比之前成熟。当前 `app_theme.dart` 已经补上了 `SegmentedButtonThemeData`，并强化了 `secondaryContainer` 等次级色彩语义，说明按钮层级治理已经起步。

这些进展说明：6.0 不应该推翻 5.0，而应该建立在这些已有成果之上，进入更细一层的视觉精修。

## 4. 当前版本仍然存在的最关键 UI 问题

### 4.1 词卡字段块仍然是“内容流式拼接”，不是“信息模块化排版”

这是第一张截图最直接暴露的问题，也是 6.0 应该放在第一优先级的点。

当前 `WordCard` 中，额外字段块是通过 `Wrap` 把若干 `Container` 平铺出来的，而这些 `Container` 没有统一宽度，也没有列数规则，更没有“全宽字段”和“半宽字段”的信息分级策略。于是界面会自然出现以下问题：

- 字段块宽度跟着内容长度漂移，导致右侧大量留白。
- 同一张词卡内，不同字段块既不像列表，也不像网格，处于一种“半流式、半堆叠”的模糊状态。
- 当上面的字段块较宽、下面的字段块较窄时，视觉节奏会被打断，用户会感到布局像是“没排整齐”。

这正是截图中红圈位置最核心的问题。

从产品设计角度看，词卡里的字段块不应该只是“把数据塞进几个容器”，而应该是一套明确的信息展示策略。比如：核心解释类字段应优先全宽展示，补充类字段可以两列排布，但必须有统一列宽和栅格对齐规则。只有这样，词卡才会从“技术上显示了字段”升级为“阅读体验舒服的内容卡片”。

### 4.2 主按钮可见性仍然不够可靠，视觉权重在不同主题下可能退化

第二张截图圈出的“空白按钮”，本质上是一个主操作失败案例。无论具体当时是什么运行态，它都说明一个问题：主操作按钮没有在任何情况下都保持足够明确的识别度。

当前主题中，普通 `FilledButton`、`FilledButton.tonal` 和 `OutlinedButton` 已经开始有分层，但还存在两个风险：

- 主按钮前景色逻辑仍然过于依赖固定分支，而不是根据真实背景亮度做对比保护。如果用户自定义了较浅的 accent 色，或者按钮退化到较浅容器背景上，文字和图标就可能变得不够明显。
- 页面里同时存在 `FilledButton`、`FilledButton.tonal`、`OutlinedButton`、`StatusBadge`、`SegmentedButton` 等多种强调控件，但“当前主动作”并没有始终拥有压倒性的视觉优先级。

这会直接导致用户在第一眼上无法判断“哪个才是当前页面的核心动作”，尤其是播放页这种高频场景。

所以 6.0 必须把“主按钮绝不失焦”当成硬规则，而不是当成一种理想状态。

### 4.3 播放页底部快捷操作区仍然像“按钮堆在一起”，而不是有层级的操作栏

第二张截图里，播放模式卡下方的快捷操作区也暴露出另一个问题：虽然按钮都在，但操作区仍然不够像一个完整的控制模块。

当前 `PlayPage` 中底部这组操作仍然是 `Wrap` 出来的若干按钮：播放、环境音、练习、词库。这样做能适配换行，但在视觉上依然偏“散”，尤其在小屏下容易出现：

- 第一行只剩一个按钮，第二行堆三个按钮，层级不均衡。
- 主按钮一旦对比度出问题，整个控制区会失去重心。
- 快捷操作区与上方的播放模式卡之间没有形成足够明确的模块边界。

也就是说，这组按钮目前更像“若干功能快捷入口”，而不像一个成熟的播放控制带。

6.0 需要把这一段从“按钮集合”升级为“操作层”，明确主操作、次操作和上下文操作之间的关系。

### 4.4 词卡信息量虽然丰富，但缺少真正的阅读优先级

第一张截图还揭示了一个更隐蔽的问题：当前词卡内的信息已经很多，包括标签、单词标题、词源、分析词义、例句、变形等，但这些信息块之间的优先级还不够清楚。

表面上看，每个模块都有卡片边框和标题，似乎已经完成了分组；但从阅读路径上看，用户仍然会感到几个问题：

- 解释型字段和补充型字段的样式过于接近，导致用户不容易快速分辨“哪个最值得先看”。
- 字段块之间的间距与尺寸不够形成节奏，有的块很大，有的块很小，但它们的视觉语言几乎一样。
- 底部控制区与字段区之间的关系还不够清楚，页面既像阅读卡，又像播放器，又像字段总览页，中心重心不够稳定。

这意味着 6.0 不只需要修“排版不齐”，还需要修“阅读优先级不够明确”。

### 4.5 当前 UI 已开始响应式，但还没有形成“页面内部栅格”

从代码角度看，当前已经做了几处响应式 fallback，这是积极变化。但这些 fallback 仍然更像“局部救火”，还没有上升到真正的页面内部栅格和布局规则。

比如现在：

- `PageHeader` 会在窄屏下让 action 下沉。
- `MiniPlayer` 会在窄屏下压缩控制区。
- `SettingTile` 会在窄屏下让 trailing 下沉。

这些都很好，但它们更多是在解决“塞不下”的问题，而不是在建立“怎么排更好看”的规则。

6.0 的目标应该是在此基础上进一步定义页面内部栅格，特别是词卡、信息块、按钮组和辅助说明文本之间的统一宽度、列数和节奏。

## 5. 6.0 的总体判断

当前项目已经跨过了“组件是否开始响应式”和“跟读是否独立化”的阶段。现在真正应该做的，是让这些已经开始变好的组件，在同一张页面上共同服从一套更严格的视觉秩序。

换句话说，6.0 的重点不再是“加更多 fallback”，而是“让页面整体一眼看上去就对”。这种“对”的标准主要包括：

- 信息块对齐
- 主次按钮清晰
- 词卡节奏稳定
- 底部控制区重心明确
- 内容块不再产生尴尬留白

所以 6.0 的定位应是：一次面向成品视觉质量的精修方案。

## 6. 方案 6.0 的核心策略

### 6.1 策略一：把词卡字段区从 `Wrap` 改造成明确的内容栅格

这是 6.0 第一优先级。

当前额外字段区不能再继续采用“内容决定宽度”的流式布局。建议重新定义字段展示规则：

- 解释类、分析类、长文本类字段统一使用全宽块。
- 短字段如变形、词性、记忆标签等才允许进入两列布局。
- 两列布局必须使用统一栅格宽度，而不是由文本内容决定卡片宽度。
- 当字段数量较少时，也不应出现一块小卡悬在左侧、右侧整片留白的情况，而应自动扩展为整行卡片。

这样可以直接解决你在第一张截图里圈出的“右侧空洞感”和“排版没收住”的问题。

### 6.2 策略二：给播放页建立真正的主操作带

当前播放页底部按钮组需要从 `Wrap` 意义上的“可换行按钮组”，升级为“主操作带 + 次操作组”。

推荐结构是：

- 主操作永远占据最明确的位置和最强的视觉对比。
- 环境音、练习、词库这些辅助动作弱化到次级样式。
- 窄屏下允许次操作换行，但主操作必须保持完整、清晰、可识别。
- 主操作区与上方播放模式卡之间建立更明确的模块间距和视觉边界。

一旦这部分重排完成，第二张截图中的“空白按钮感”和“操作区散乱感”会明显下降。

### 6.3 策略三：建立按钮对比度保护规则

6.0 需要正式引入按钮对比保护，而不是继续依赖某个颜色大致可用。

建议至少建立三条规则：

- 主按钮前景色必须根据背景亮度自动选择深色或浅色，而不是只用固定白色/固定深色。
- 当 accent 色接近高亮浅色时，主按钮应自动压深背景色或使用更高对比容器色。
- `FilledButton.tonal` 只用于明确的次级强调，不得让它在视觉上与主按钮产生混淆。

这会直接解决“按钮颜色不清”以及“按钮像空白块”的问题。

### 6.4 策略四：让词卡中的阅读和操作彻底分层

当前词卡同时承担阅读、对象状态和操作入口三件事，但这三层在视觉上还没有彻底分开。6.0 建议明确拆分为：

- 顶部：对象状态层，例如任务词、收藏标签。
- 中部：内容阅读层，例如词义、词源、例句、分析。
- 底部：对象操作层，例如播放发音、跟读、收藏、任务管理。

一旦这三层被分清，词卡的重心会稳定很多，页面也会更有“成品感”。

### 6.5 策略五：把“响应式 fallback”升级为“页面内部布局规则”

5.0 已经让多个组件开始能在小屏下不崩，但 6.0 要求更高：不能只满足“塞得下”，而要满足“排得好”。

建议正式定义以下规则：

- 主页面首屏模块之间的最小/标准/宽松间距。
- 卡片内部标题、正文、辅助说明、按钮组的固定节奏。
- 小屏下按钮是否换行、是否纵向堆叠、何时独占整行。
- 内容块在 1 列与 2 列之间的切换条件。

这会让界面不只是“遇到窄屏就临时下沉”，而是从一开始就按照移动端页面栅格来组织。

## 7. 重点模块升级建议

### 7.1 `WordCard`

`WordCard` 是本轮截图问题的中心。6.0 建议它成为本轮精修的核心模块。

优先级最高的事情包括：

- 重做字段卡布局，废除当前的内容自适应宽度流式排版。
- 明确长字段与短字段的展示级别。
- 统一字段卡的高度节奏和间距。
- 调整底部操作区的主次关系，让“跟读”与“播放”不再只是并列按钮，而是符合场景优先级。

### 7.2 `PlayPage`

播放页当前的问题不是功能少，而是页面重心还不够稳。建议重点处理：

- 底部快捷操作区的主次重排。
- 主播放按钮的可见性保护。
- 上下模块之间的视觉边界增强。
- 当前词卡与控制层之间的关系再明确一些，避免内容阅读区和控制区相互打架。

### 7.3 `MiniPlayer`

当前 `MiniPlayer` 已经开始具备展开层，这是对的。6.0 不需要推翻它，而是要继续压缩“表层按钮数量”和“默认展示复杂度”，让默认态更稳、更清楚，复杂控制继续留在展开层。

### 7.4 `WordRow` 与 `LibraryPage`

这一部分当前已经比 5.0 有明显进步，但仍建议继续强化：

- `WordRow` 的主动作按钮继续收口，避免重新膨胀。
- 跟读入口的按钮视觉权重要和列表浏览节奏相匹配，不能喧宾夺主。
- 词库页继续保持定位稳定，但要避免条目在长文本条件下再次出现结构失衡。

### 7.5 `app_theme.dart`

主题层在 6.0 中的重点不是继续加更多颜色，而是建立稳态规则：

- 主按钮对比保护
- 次级按钮统一语义
- 状态块和选中态清晰化
- 亮浅主题下的主按钮安全色策略

只要这一层稳定了，很多页面里的“按钮看起来像空白块”问题就会自然下降。

## 8. 6.0 分阶段实施建议

### 阶段 A：截图问题直修

优先修 `WordCard` 字段区布局和 `PlayPage` 主操作区，把当前截图中最明显的问题先彻底处理掉。

### 阶段 B：按钮系统稳态化

统一主按钮、tonal 按钮、outlined 按钮和 segmented 状态按钮的视觉权重，加入对比保护规则。

### 阶段 C：词卡与页面内部栅格建立

把当前“组件各自响应式”进一步升级为“页面内部统一栅格”，特别是词卡、信息块、按钮组这类复合结构。

### 阶段 D：播放页成品感收口

把播放页真正做成一个视觉上稳定、主次分明、随时可用的主场景页。

### 阶段 E：全局视觉细节精修

最后再统一帮助页、设置页、复习页和更多页中的卡片节奏、按钮表现和辅助文案层级，让全产品观感进一步收口。

## 9. 6.0 验收标准

从视觉层看，6.0 完成后应满足以下判断：

- 词卡字段区不再出现截图中这种明显的右侧空洞和块状错位。
- 主按钮在任何常用主题下都不会出现“像空白按钮”或“看不清文字”的情况。
- 播放页底部快捷操作区有明确重心，不再像临时堆叠的按钮集合。
- 页面内部的信息块有稳定节奏，阅读和操作层清晰分离。

从交互层看，6.0 完成后应满足以下判断：

- 用户可以一眼识别当前页面最推荐点击的按钮。
- 跟读虽然已成为单词级模式，但入口不会破坏页面整体秩序。
- 列表、词卡和控制区在小屏上都保持稳定，不会给人“排版松散”的感觉。

从产品层看，6.0 完成后应满足以下判断：

- 当前 UI 不只是结构上完整，而且观感上开始接近真正成品。
- 截图中这类“看起来哪里不太对”的问题明显减少。
- 产品从“组件已经能工作”迈向“界面本身具有可信度和完成度”。

## 10. 最终建议

当前项目已经证明，Flutter 重构的方向是成立的，5.0 阶段的大部分基础治理也确实已经开始见效。现在的问题，已经不再是没有方向，而是到了必须认真打磨视觉秩序的时候。

你这次给的两张截图非常关键，因为它们都不是功能 bug，而是“成品感 bug”。这类问题往往不会让程序报错，但会极大影响用户对产品成熟度的判断。

因此，6.0 不建议再继续发散到大量新功能，而应明确投入到以下两件事情上：

第一，把词卡和播放页这种最常被看见的核心界面做成真正稳定、顺眼、对齐的移动端成品。  
第二，把按钮层级和局部布局规则做成不会轻易退化的系统规则。

如果 5.0 的重点是“响应式和单词级操作开始成立”，那么 6.0 的重点应该是：

让这些已经成立的能力，在真实界面里真正看起来对、排得稳、主次分明。
