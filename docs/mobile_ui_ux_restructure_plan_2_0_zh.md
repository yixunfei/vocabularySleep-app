# Vocabulary Sleep Flutter UI/UX 重构方案 2.0

更新时间：2026-03-10

## 1. 版本说明

本文件是在第一版《移动端 UI/UX 重构方案》的基础上，结合当前项目已经完成的 UI 改造结果，重新评估后形成的 2.0 版本方案。

相比 1.0，当前项目已经发生了实质性变化：

- 应用入口已经从单页 `HomePage` 迁移为 `AppShell + NavigationBar + IndexedStack` 结构。
- 主界面已经拆分出 `播放`、`词库`、`练习`、`更多` 四个一级页面。
- 全局主题已经从首页局部驱动转为应用级 `buildAppTheme(state.config.appearance)`。
- 视觉层已经引入 `Sleep / Focus` 双体验模式。
- `BusyOverlay`、`AppBackground`、`MiniPlayer`、`WordCard`、`WordRow` 等基础组件已经建立。
- `设置中心`、`词本管理`、`数据管理`、`跟读页`、`词条详情页` 等页面已经从原巨型页面中拆出。

因此，方案 2.0 的目标不再是“从单页大控制台拆成多页”，而是进入下一个阶段：把当前已经成型的过渡架构，收敛为一套稳定、完整、可持续演进的移动端产品架构。

## 2. 当前版本重新评估

### 2.1 当前版本的积极变化

这轮 UI 改造是有效的，而且已经完成了最关键的第一步：移动端产品骨架已经建立。

首先，应用已经具备真正的壳层结构。当前 `main.dart` 已经不再直接将首页挂为旧版 `HomePage`，而是通过应用级主题和 `AppShell` 作为统一入口，这一点非常关键，说明项目的 UI 架构已经从“页面驱动”转向“应用壳层驱动”。

其次，一级导航结构已经明确。`AppShell` 中的四个页面分别对应 `PlayPage`、`LibraryPage`、`PracticePage`、`MorePage`，并通过底部 `NavigationBar` 串联，同时用 `IndexedStack` 保持页面状态，这使得移动端主路径已经具备稳定承载能力。

第三，设计语言开始成体系。当前已经存在 `AppThemeTokens`、`AppBackground`、`PageHeader`、`SectionHeader`、`SettingTile`、`WordCard` 等基础部件，说明设计系统已不再依赖原有大页面内部的局部拼装。

第四，详情页的阅读态优先逻辑已经落地。`WordDetailPage` 将编辑和删除收入口袋菜单，而将阅读内容作为主结构保留，这是非常符合移动端内容产品习惯的变化。

第五，全局忙碌态已经开始被统一处理。当前 `AppState.busy` 已经通过 `BusyOverlay` 在壳层统一接入，这比过去仅靠局部 `SnackBar` 反馈明显更进一步。

从产品演进阶段来看，当前项目已经完成了“从 0 到 1 的架构拆分”，进入“从 1 到 1.5 的收口阶段”。

### 2.2 当前版本最值得肯定的部分

当前项目最值得肯定的，不是单个页面多漂亮，而是以下三个方向已经判断正确并开始实现：

第一，默认主结构已经移动端化。四个一级 tab 已经建立，这使得未来无论再加练习流程、主题工作台还是数据能力，都有稳定容器。

第二，体验模式已经产品化。`Sleep / Focus` 不再只是配色差异，而是带有明确产品语义的体验层。这个方向是对的，说明项目已经不再把主题仅仅理解成“好看不好看”，而是把它看作面向场景的默认策略。

第三，页面职责已经明显分离。播放页主要服务听词，词库页主要服务搜索与浏览，练习页主要服务测试与跟读，更多页主要服务管理入口。这为后续优化打下了正确基础。

### 2.3 当前版本仍然存在的核心问题

虽然 1.0 阶段的目标基本达成，但当前版本仍然显然是“过渡态”，还不能算真正完成的移动端产品版。

#### 问题一：新架构和旧架构仍然并存

当前 `MorePage` 仍然公开保留了“高级工作台 / Classic workspace”入口，并直接跳回旧版 `HomePage`。这说明新结构和旧结构目前是双轨并行的，而不是已经完全迁移完成。

这种双轨状态短期有价值，因为它可以降低重构风险；但如果长期保留，会带来三个问题：

- 用户心智被分裂，不知道新页面和旧工作台哪个才是正式入口。
- 设计语言无法真正统一，因为旧页面与新页面不是同一套体验逻辑。
- 工程维护成本持续增加，后续功能很容易在新旧两套结构中发生漂移。

换句话说，当前最大的系统性问题不再是“没有移动端架构”，而是“移动端新架构已经出现，但还没有完全取代旧架构”。

#### 问题二：设置中心仍是过渡版，而不是最终版

当前 `SettingsPage` 已经开始做体验模式、基础播放、语音、识别和界面细节设置，但它自己也明确承认“更深的高级项仍保留在旧工作台”。

这意味着当前设置中心仍是“轻量入口”，不是完整的正式设置系统。它解决了普通用户的第一层需求，但没有完成高级能力迁移，也没有真正建立清晰的分层结构。

因此当前项目的问题不是“设置太多”，而是“设置的完整架构还没收敛完成”。

#### 问题三：练习页已经独立，但还没有形成完整学习闭环

当前 `PracticePage` 已经将测试模式和跟读移出了首页，这是正确的，但它本质上仍然围绕“当前词”展开，尚未形成完整的练习会话模型。

目前它更接近“当前词练习面板”，而不是“练习系统”。缺少的关键能力包括：

- 练习范围选择
- 会话进度管理
- 错词或弱词回看
- 连续练习结果反馈
- 今日练习或分组练习入口

这意味着 2.0 阶段需要把练习页从“功能承接页”升级为“完整场景页”。

#### 问题四：文案与国际化体系仍然是过渡方案

当前新 UI 使用了 `ui_copy.dart` 作为轻量文案适配层，这对快速推进重构非常有帮助，但它也带来了新的问题：应用可见文案开始同时分散在 `ARB` 和 `ui_copy.dart` 两个体系中。

这会导致：

- 文案维护入口不统一
- 语言扩展成本上升
- 页面文案风格容易前后不一
- 局部硬编码文案难以统一校对

此外，像 `BusyOverlay` 里的 `Working...`、设置页中直接显示 `provider.name` 这类实现，也说明当前文案与交互细节仍有明显收口空间。

#### 问题五：产品完成度还停留在“结构成形”，尚未进入“精修成品”

当前页面已经有正确的划分，但很多地方仍带有明显的过渡感，例如：

- `MiniPlayer` 已经存在，但还是一个平面迷你控制条，没有形成更完整的“正在播放层”。
- `LibraryPage` 已经具备搜索与字母跳转，但仍有部分交互是工程化快速实现，例如用固定高度估算滚动定位。
- `FollowAlongPage` 已经独立，但高级引擎选择仍直接暴露在主页面，不够分层。
- `MorePage` 中仍包含“当前状态 + 管理入口 + 旧工作台入口”的混合逻辑，说明它还是一个过渡汇总页。

所以 2.0 的重点，不是继续证明“可以拆页”，而是开始做真正的产品收口。

## 3. 2.0 的总体判断

方案 2.0 的核心判断如下：

当前项目已经完成了“结构拆分”，下一阶段应该转入“能力收敛 + 体验收口 + 旧架构退场”。

这意味着 2.0 不应再以“大规模重新设计全部页面”为目标，而应以以下三件事为主：

1. 补齐新结构中仍然缺失的正式页面和正式设置层。
2. 把旧 `HomePage` 中仍有价值的深度能力逐步迁移出来。
3. 在迁移完成后，去掉用户可见的旧工作台入口，统一产品心智。

因此，2.0 是一个“收口型方案”，而不是“推翻型方案”。

## 4. 2.0 目标

### 4.1 产品目标

- 让新架构成为唯一正式产品路径。
- 让 `播放 / 词库 / 练习 / 更多` 四个一级入口真正完成职责闭环。
- 让旧版 `HomePage` 退出用户主路径，只在开发或临时兼容阶段保留。
- 让设置中心不再依赖“旧工作台兜底”来承载高级能力。

### 4.2 体验目标

- 让播放页成为真正稳定、舒服、可信的默认首页。
- 让练习页从“当前词功能页”升级为“可连续使用的练习场景”。
- 让更多页从“过渡汇总页”升级为“清晰的管理与配置中心”。
- 让文案、命名、按钮和状态表达更统一，减少“开发态痕迹”。

### 4.3 工程目标

- 停止继续向旧 `HomePage` 新增功能。
- 将新 UI 中剩余的高级功能页面化。
- 清理新旧架构双轨状态。
- 建立更统一的 UI copy 和主题配置路径。

## 5. 方案 2.0 的核心策略

### 5.1 策略一：从“双轨并行”转向“新架构唯一正式入口”

2.0 最重要的一条策略是：旧 `HomePage` 不再作为对普通用户公开的常规入口。

建议执行方式如下：

- 第一步，将 `MorePage` 中的“高级工作台”入口从常规卡片里移除。
- 第二步，在迁移未完成阶段，仅通过 debug 开关、开发菜单或内部构建渠道保留旧工作台入口。
- 第三步，当高级设置和深度能力迁移完成后，彻底将旧 `HomePage` 标记为 legacy，仅作为参考代码保留，或继续拆解后删除。

这一步对产品心智统一意义非常大。

### 5.2 策略二：把“高级能力”从旧工作台迁入正式页面体系

当前旧工作台之所以仍被保留，原因不是新架构方向不对，而是高级能力仍未完全迁出。2.0 应将这部分能力拆成正式的新页面，而不是继续压在旧工作台里。

建议新增或强化以下页面：

- `VoiceSettingsPage`
- `RecognitionSettingsPage`
- `AppearanceStudioPage`
- `PlaybackAdvancedPage`
- `PracticeSessionPage`

当前 `SettingsPage` 保留为一级设置中心，但只承担概览和常用项；深度项进入二级页，而不是回退到 legacy 首页。

### 5.3 策略三：把练习页从“当前词练习”升级为“练习系统”

练习页是 2.0 必须增强的主战场。

当前 `PracticePage` 已经建立了方向，但下一步应补齐：

- 练习入口层：当前词、当前词本、任务词、错词回看。
- 练习会话层：连续练习、进度条、会话结束反馈。
- 练习结果层：正确率、连续次数、需要加强词。

这会让练习页真正从“辅助功能页面”变成与播放页、词库页并列的核心产品场景。

### 5.4 策略四：让设置中心真正成为正式产品层，而不是半过渡入口

设置中心 2.0 建议拆为以下层级：

- 一级：设置首页
- 二级：播放与语音
- 二级：识别与练习
- 二级：外观与主题
- 二级：语言与通用

其中设置首页只做概览和分发，不堆太多复杂控件。每一类高级能力都有自己的正式页面，不再由一个大页承载全部复杂表单。

### 5.5 策略五：内容系统和命名体系统一

2.0 必须开始清理文案和命名体系。建议建立统一规则：

- 页面级正式文案进入 `ARB`。
- `ui_copy.dart` 只保留页面间共享的过渡文案，后续逐步收敛到 `ARB`。
- 所有对用户可见的 provider、mode、engine 名称都必须经过映射，不直接显示枚举值。
- 所有全局状态文案，包括 Busy、错误、成功提示，都必须纳入统一国际化系统。

这是 2.0 非常值得投入的一部分，因为这会直接影响产品的成熟度观感。

## 6. 页面升级方案

### 6.1 播放页 2.0

当前播放页已经具备较好的基础结构，2.0 的重点不是推翻，而是增强：

- 保留当前 `PageHeader + WordbookSwitcher + WordCard + 播放模式 + 快捷操作` 的结构。
- 为 `MiniPlayer` 增加可展开能力，支持“上滑查看更多播放控制”。
- 增加 `Sleep Timer` 或“定时停止”能力，纳入播放工具层。
- 增加“当前播放队列”或“即将播放”轻量入口。
- 将环境音入口从单次按钮升级为“常驻的播放工具面板成员”。

播放页 2.0 的目标是让它从“结构正确的首页”升级为“真正舒服的默认首页”。

### 6.2 词库页 2.0

词库页当前已经达到了 1.0 阶段目标，但 2.0 可以继续优化以下方面：

- 增加 `全部 / 收藏 / 任务` 等快速范围筛选。
- 优化字母跳转交互，让它更贴近移动端单手操作。
- 为搜索结果和当前词跳转建立更精确的滚动定位逻辑。
- 增加列表空状态与导入引导之间的视觉连接。
- 在词条详情页中补充“进入练习”“快速编辑”“快速复制”等更明确的上下文动作。

词库页的关键词不是“更复杂”，而是“更顺手”。

### 6.3 练习页 2.0

练习页 2.0 建议分成两个层级：

- 练习首页：选择练习方式。
- 会话页：连续练习。

练习首页建议至少有四个入口：

- 当前词练习
- 当前词本练习
- 任务词练习
- 跟读练习

会话页建议统一具备：

- 当前题目
- 当前进度
- 会话范围
- 下一题逻辑
- 本次会话反馈

对于 `FollowAlongPage`，建议在 2.0 中引入“普通模式 / 高级模式”区分：

- 普通模式只保留参考发音、录音、结果。
- 高级模式才展开 provider 和引擎细节。

### 6.4 更多页 2.0

更多页当前是管理入口集合页，这个方向是对的，但它还需要去掉过渡态痕迹。

2.0 中的更多页应明确只承担两件事：

- 管理类入口
- 设置类入口

建议最终保留以下主分组：

- 设置中心
- 词本管理
- 数据管理
- 关于与帮助

“当前状态”卡片可以弱化或改造成更明确的“会话快照”，而“高级工作台”入口应从正式用户路径中移除。

### 6.5 设置中心 2.0

设置中心建议从当前单页，进一步拆为：

- `SettingsHomePage`
- `VoiceSettingsPage`
- `RecognitionSettingsPage`
- `AppearanceSettingsPage`
- `LanguageSettingsPage`

`SettingsPage` 当前保留的体验模式切换方向是正确的，应继续强化为最上层入口之一。

但以下内容需要补做：

- TTS provider 的本地化显示和分组说明
- ASR provider 的本地化显示和使用说明
- 高级项入口页，而不是继续依赖旧工作台
- 页面级完成提示与配置变更反馈

### 6.6 词本管理与数据管理

当前这两页已经具备正式页面形态，2.0 阶段主要做的是精修而不是重构：

- 词本管理页增加分组视图，例如系统词本 / 特殊词本 / 自定义词本。
- 合并流程可以从对话框升级为更清晰的步骤式表单，尤其是在移动端。
- 数据管理页增加说明文案和操作后反馈。
- 关键破坏性操作增加更明确的后果说明和结果反馈。

## 7. 视觉系统 2.0

### 7.1 当前视觉判断

当前视觉系统已经从过去的杂糅状态提升为“有统一背景、有统一 token、有体验模式区分”的结构，这是明显进步。

但 2.0 需要进一步收口，不再把重点放在“能调很多参数”，而是放在“默认就是成熟的”。

### 7.2 视觉策略

2.0 推荐继续保留 `Sleep / Focus` 双模式，但将定位进一步明确：

- `Sleep`：默认首页更弱刺激，更少可见字段，更突出持续播放。
- `Focus`：默认更高信息密度，更强调搜索与练习。

同时需要控制以下风险：

- 不再让实验性特效作为默认体验出现。
- 不再让高级视觉参数直接干扰普通用户的主路径。
- 保持新页面与旧页面视觉逻辑完全隔离，逐步消灭旧工作台的视觉干扰。

### 7.3 组件统一

2.0 应继续沉淀和统一下列组件：

- `PageHeader`
- `WordCard`
- `WordRow`
- `SettingTile`
- `MiniPlayer`
- `SectionHeader`
- `StatusBadge`
- `BusyOverlay`

所有新增页面优先复用这些组件，不再出现“新页面重新造一套列表样式”的情况。

## 8. 技术实施方案 2.0

### 8.1 架构原则

2.0 的技术实施原则如下：

- 不再向旧 `HomePage` 添加新功能。
- 新能力只进入新页面结构。
- 迁移旧能力时优先拆到独立页面或独立 sheet。
- 让 `AppShell` 成为唯一正式产品入口。

### 8.2 建议新增的页面与文件

建议新增或进一步拆分以下文件：

```text
lib/src/ui/pages/
  settings_home_page.dart
  voice_settings_page.dart
  recognition_settings_page.dart
  appearance_studio_page.dart
  playback_advanced_page.dart
  practice_session_page.dart
  review_session_page.dart
  sleep_tools_page.dart
```

如需更轻量，也可以先将其中部分实现为 `bottom sheet`，但它们应被视为正式页面能力的一部分，而不是临时替代旧工作台的入口。

### 8.3 文案与国际化收敛

建议新增一轮专门的文案治理任务，内容包括：

- 清理所有面向用户的硬编码英文。
- 清理所有直接显示 `enum.name` 的界面。
- 评估 `ui_copy.dart` 中哪些文案要迁入 `ARB`。
- 对新 UI 页面的中文、英文进行统一语气和术语规范。

### 8.4 Busy 与反馈升级

当前 `BusyOverlay` 已经存在，但 2.0 建议增强为可传入消息版本，例如：

- 导入中
- 导出中
- 合并中
- 下载模型中
- 处理中

这样比统一显示 `Working...` 更符合产品完成度，也更利于国际化。

### 8.5 Legacy 退场计划

建议明确一条技术约束：

在 2.0 实施期间，所有仍需依赖旧 `HomePage` 的功能都要建立迁移清单；迁移清单清空后，`MorePage` 中移除旧工作台入口。

这是一个必须被显式管理的目标，不能无限期拖延。

## 9. 分阶段实施计划

### 阶段 A：收口准备

目标：

- 冻结旧 `HomePage` 的新增功能。
- 盘点旧工作台仍未迁出的能力。
- 建立迁移清单。

重点产出：

- Legacy 功能盘点表
- 2.0 页面迁移任务列表

### 阶段 B：设置与高级能力迁移

目标：

- 将旧工作台中的高级播放、语音、识别、外观能力拆到正式页面。
- 让 `SettingsPage` 不再只是过渡入口。

重点产出：

- `VoiceSettingsPage`
- `RecognitionSettingsPage`
- `AppearanceStudioPage`
- `PlaybackAdvancedPage`

### 阶段 C：练习系统升级

目标：

- 将练习页从当前词功能页升级为会话型学习页。
- 建立连续练习、错词回看、任务词练习路径。

重点产出：

- `PracticeSessionPage`
- `ReviewSessionPage`
- 统一练习流程入口

### 阶段 D：播放器与播放页增强

### 阶段 E：Legacy 退场与产品精修

- 统一文案、按钮命名、提示语、空状态、错误态。

重点产出：

- More 页正式版
- 设置中心正式版

## 10. 2.0 验收标准

### 产品层

- 普通用户不再需要进入旧工作台才能完成常见配置。
- 四个一级页面都具备清晰职责，不再只是分流旧首页功能。
- 练习页具备独立会话价值，而不是仅围绕当前词。
- 更多页不再承担“新旧架构中转站”角色。

### 体验层

- 新用户从首页开始可以在 3 秒内理解主路径。
- 所有高频操作在新结构内完成，无需回退 legacy。
- 所有用户可见文本均本地化、命名一致。
- 忙碌态、错误态和完成态表达清晰。

### 工程层

- `flutter analyze` 持续通过。
- `flutter test` 持续通过。
- 旧 `HomePage` 不再新增正式能力。
- 高级能力已完成迁移清单的大部分或全部。

## 11. 最终建议

当前项目已经跨过了最难的一步：它不再是单页巨石结构，而是已经具备移动端产品骨架。

因此，2.0 不建议再做一次“推倒重来”的全面设计，而应明确进入“新结构收口、旧结构退场、产品完成度提升”的阶段。

如果要用一句话概括 2.0，那么应该是：

从“新架构已经成立”走向“新架构成为唯一正式产品”。

这将是当前项目最有价值、也最符合阶段收益的下一步。
