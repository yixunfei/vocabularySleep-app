# Vocabulary Sleep Flutter UI/UX 重构反馈方案 10.0

更新时间：2026-03-11

## 1. 版本定位

10.0 不是对 9.0 的简单重复，而是基于当前项目“已经再次完成部分修改”后的重新评估版本。与前几轮最大的不同在于，当前项目已经不再处于“关键能力缺失”的阶段，而是正式进入“系统开始成形，但需要收口与稳定”的阶段。

换句话说，现在的问题已经不是“有没有索引”“有没有回到顶部”“有没有大词库优化”“有没有播放中的单词切换”，而是这些能力放在一起后，是否形成了一套真正稳定、清晰、顺手、低负担的移动端体验。  
这也是 10.0 的核心判断：当前项目已经明显从“补能力”走向“做系统”，而 10.0 的任务，就是帮助它从“能用的功能集合”走到“成熟的产品体验”。

## 2. 本轮复评依据

本轮重新检查了以下关键文件的当前实现状态：

- `lib/src/ui/pages/library_page.dart`
- `lib/src/ui/pages/play_page.dart`
- `lib/src/ui/widgets/word_card.dart`
- `lib/src/ui/widgets/word_row.dart`
- `lib/src/ui/app_shell.dart`
- `lib/src/ui/theme/app_theme.dart`
- `lib/src/state/app_state.dart`

同时对照了已有文档基线，尤其是 9.0 所强调的几个问题：词库惰性化、导航触达、搜索性能、播放页对象直达、浮层秩序与验证覆盖。

从当前代码结果看，项目已经在多个关键点上落实了 9.0 里建议推进的内容：  
词库页已经切换到 `CustomScrollView + SliverList`，不再停留在纯 `ListView(children)` 的阶段；  
搜索已经接入前端防抖；  
状态层已经加入 `visibleWords` 缓存；  
滚动中的回顶按钮与索引按钮都已经存在；  
索引弹层已经支持“仅可用字母”与“展开全 A-Z”双态切换；  
播放页已经新增上一词、下一词与滑动切词。

这说明当前项目不是停滞不前，而是在持续朝正确方向推进。  
但也正因为这些能力都开始出现了，新的问题也变得更具体了：路径开始重复、状态开始分层、行为之间开始彼此影响。10.0 的重点，正是处理这些“第二阶段问题”。

## 3. 相比 9.0，当前项目已经完成的关键进展

### 3.1 词库页终于进入了真正的惰性结构阶段

这是本轮最值得肯定的变化之一。  
当前词库页已经不再是简单的整页 children 拼接，而是转入 `CustomScrollView` 结构，并把词条列表切换为 `SliverList`。这意味着项目已经开始以“长列表产品”的思路来组织页面，而不是继续用普通内容页硬撑海量词库。

虽然当前仍保留了分页窗口 `_visibleItemCount` 的概念，但相比早期版本，这已经是一个更合理的方向：首屏只展示部分结果，滚动接近底部后继续增长，而不是一开始就把所有结果都压到界面上。这为后续真正的长列表性能治理打下了基础。

### 3.2 搜索链路开始具备“减压意识”

当前 `LibraryPage` 已经引入了 `160ms` 的输入防抖，并通过 `_onSearchChanged` 与 `_commitSearchQuery` 把“输入中的文本”和“真正提交到状态层的查询值”分开。  
同时，`AppState.visibleWords` 也已经增加缓存机制，不再是每次读取都盲目重算。这是 9.0 里非常重要的一条建议，如今已经开始落地。

这类优化用户不一定会直接“看见”，但会明显影响体感。它标志着产品开始从“页面怎么摆”进入“状态怎么流动”的阶段，是产品成熟的重要信号。

### 3.3 索引入口已经从单一横向条带升级成双层入口

当前词库页的索引系统已经不再只有顶部的一条横向字母条。  
现在项目已经同时具备：

- 顶部的索引工具卡片
- 底部弹出的完整字母索引面板
- 深滚后出现的悬浮索引按钮
- 前缀跳转作为精确定位入口

这说明“索引”已经开始从一个控件变成一个系统。  
这一步非常重要，因为它标志着词库页正在摆脱“只有一个入口，用户必须自己会用”的阶段，转而进入“不同滚动状态下都有相应导航工具”的阶段。

### 3.4 回到顶部已经从页面内容按钮升级为真正悬浮锚点

相比第八版以前那个停留在内容流中的“回到顶部”按钮，当前项目已经把回顶能力做成了滚动后出现的悬浮按钮。这是非常正确的方向。  
它说明团队已经开始理解：真正的长列表回顶，不应该摆在用户最不需要它的地方，而应该在用户已经滚深时自动出现。

这一步虽然看起来只是按钮位置变化，但本质上是交互模型的变化。它代表项目开始从“有功能”走向“功能出现在正确时机”。

### 3.5 播放页已经具备更连续的切词体验

当前 `WordCard` 已新增：

- 上一个词 / 下一个词按钮
- 单词区域左右滑动切词
- 播放状态下的暂停状态标识

这意味着播放页终于不再只是“单词展示 + 播放按钮”的静态页，而是开始具备连续学习流中的节奏感。  
从实际体验上看，这一步会明显降低用户在播放过程中来回跳回词库选择下一个词的需求。

## 4. 当前版本最值得重视的实际体验问题

### 4.1 搜索输入状态与结果状态已经分层，但还没有完全统一

这是当前项目非常典型的“第二阶段问题”。  
搜索防抖的接入本身是对的，但现在也带来了一个新的体验结构：输入框里的文本和 `AppState.searchQuery` 在短时间内并不是同一个状态。

这会造成几个潜在问题：

第一，用户已经在输入框里看到了新文本，但词库结果还没有同步刷新。这个 160ms 的延迟本身不算大，但如果此时用户继续做别的动作，例如切换搜索模式、打开词本切换弹层、触发某个跳转动作，就会进入“界面展示的是 A 状态，但状态层实际还是 B 状态”的短暂混合期。

第二，当前 `SegmentedButton` 切换搜索模式时，并不会先强制提交尚在 debounce 中的查询值。这意味着用户在快速输入后立刻切换搜索模式，有机会出现一次“先按旧 query 切模式，再按新 query 刷结果”的双跳感。

第三，这种状态分层虽然提高了性能，但也对产品一致性提出了更高要求。10.0 需要把它正式视为“搜索状态管理设计”的一部分，而不是仅仅看成一个性能优化小技巧。

这不是说当前实现错误，而是说明它已经迈入“需要明确产品规则”的阶段。  
10.0 应该明确：搜索输入、搜索提交、搜索模式切换、词本切换之间的优先级和同步时机。

### 4.2 词库页现在已经有两个索引入口层，但开始出现“入口重复”

当前项目里，索引工具已经不再缺失，反而开始接近“重复”：

- 首屏有“索引工具”卡片里的“展开索引”按钮
- 深滚后还有一个悬浮的索引按钮

这说明项目已经从“索引入口不够”进入“索引入口怎么协同”的新阶段。  
如果处理不好，用户会开始感受到一种 subtle 的混乱：同一个动作为什么在两个地方出现？它们是同级的吗？我该记住哪个？是不是以后还会有第三个？

从产品设计角度看，这并不意味着要删掉其中一个，而是意味着需要开始定义它们的角色：

- 顶部索引入口更适合承担“首次浏览”和“结果解释”
- 悬浮索引入口更适合承担“深滚后快速重定位”

如果不做这层角色定义，它们现在虽然都可用，但会慢慢退化成“重复入口堆叠”，而不是“分场景入口协同”。

### 4.3 词库工具区已经更完整，但仍然不是真正的 sticky 工具层

当前词库页虽然已经切到 `CustomScrollView`，但顶部的搜索区、索引工具区、分组模式区仍然属于普通滚动内容，而不是可吸附工具层。  
现在的悬浮索引按钮解决了一部分“深滚后找不到索引入口”的问题，但仍然没有解决“深滚后重新调整搜索/筛选/模式”的问题。

也就是说，当前项目已经通过悬浮按钮补上了“索引再触达”，但还没有真正建立“滚动中工具层”。  
在一个高频使用的词库产品中，这两者并不完全等价。

10.0 应该把这个问题提升为结构级问题：  
词库页不只是需要一个悬浮索引按钮，而是需要逐步形成一个能够在滚动中持续提供控制能力的工具层，哪怕一开始只是部分 sticky，而不是全部 sticky。

### 4.4 分页加载已经开始生效，但当前策略更像“递增展示”，还不是“长期稳定窗口”

当前 `_visibleItemCount` 会随着滚动不断向上增长，但并不会在长时间浏览中回收前面已经加载的可见窗口。  
这意味着，当前实现虽然避免了首屏一次性渲染所有词条，但在长时间使用后，页面仍然会逐步积累更多可见条目，最终再次走向“越来越重”的状态。

从这个角度看，当前版本更准确地说是“渐进式加载”，而不是严格意义上的“窗口化长列表”。  
它已经比早期版本好很多，但在超大词库、长时间滚动、频繁来回跳转的真实场景里，10.0 仍然建议继续前进一步：从“只增长不回收”的可见数策略，走向更成熟的“分段锚点 + 惰性构建 + 必要时软回收”。

这件事不一定要立刻做得很重，但在 10.0 里至少应该被正式定义为下一阶段的问题，而不是认为当前分页已经彻底解决了大词库性能。

### 4.5 字母索引已经支持可用字母/全量字母双态，但结果语境仍可继续收紧

这一点相比第八版已经明显更成熟了。  
当前索引弹层在结果较少时支持“仅显示可用字母”，并允许用户再手动展开为全量 A-Z，这已经比之前强很多。

但从产品体验上看，仍然还有进一步收紧的空间：

第一，当前“当前仅 X 个结果，已自动简化字母索引展示”的说明文字是对的，但它还是以“告诉用户系统做了什么”为主，而不是以“告诉用户现在最应该做什么”为主。  
也就是说，它还是偏系统解释，而不是操作引导。

第二，当结果集很小的时候，页面最有价值的动作通常已经不再是字母跳转，而是“清除搜索”“切换词本”“直接点结果进入详情”。现在项目已经加了 `Clear search`，这是好事，但整体工具区仍然略偏“索引优先”，还可以再向“结果优先”移动一步。

10.0 应该把这件事定义清楚：  
短结果集场景下，索引存在即可，不需要继续占据太高权重。

### 4.6 播放页已经能切词，但“当前词仍不是第一视觉中心”

这是当前播放页里最值得继续打磨的问题。  
现在 `WordCard` 的结构已经比以前完整很多，但从视觉顺序上看，顶部先出现的是控制区，再出现单词本体内容。  
这意味着，当前播放页虽然在功能上已经具备“围绕当前词进行操作”的能力，但在视觉心智上，用户第一眼看到的仍更像是一组控制按钮，而不是一个“正在被学习的词对象”。

这会带来两个后果：

第一，当前词虽然存在，但不够像“主角”。  
第二，跟读入口虽然存在，但依然更偏按钮级，而不是对象级。

10.0 建议把这一点明确列为体验问题，而不是继续只从功能角度看待它。  
对于学习产品来说，“当前词是主角”这件事比“上一个词/下一个词按钮在哪里”更重要。

### 4.7 当前词仍然没有形成“点击对象即进入动作”的主路径

第九版里已经指出过，播放页应该围绕当前词建立对象级入口。  
从当前代码看，这件事仍未彻底完成。现在播放页主要增加了滑动切词和上一词/下一词，但“点击当前词主体本身进入跟读或动作层”仍不是主路径。

这意味着，当前项目已经改善了“词与词之间如何移动”，但还没有真正完善“对这个词本身如何深入操作”。  
这是一个非常典型的阶段差异：  
导航变顺了，但对象交互还没完全成型。

10.0 建议明确把它列为播放页下一阶段的第一优先级，而不是继续优先增加更多按钮和更多快捷动作。

### 4.8 测试体系仍然滞后于当前 UI 复杂度

当前项目已经有 `AppState` 逻辑测试，也有基础 sanity 和 settings 测试，这是好基础。  
但从当前 UI 复杂度看，词库页已经不再是简单静态列表，而是包含：

- 搜索防抖
- 可见窗口增长
- 深滚悬浮回顶
- 深滚悬浮索引
- 索引弹层双态切换
- 词本切换
- 跳转后滚动定位

这类页面如果继续缺少 widget 层和 integration 层验证，很容易出现一种情况：  
每次改一处都能通过 analyze 和纯逻辑 test，但真实路径已经悄悄退化。

因此，10.0 必须把测试覆盖视为“体验系统的一部分”，而不是开发完成后的附属工作。

## 5. 10.0 的总体判断

当前项目已经明显迈过了最难的起步阶段。  
从整体状态看，它不再像一个“正在试着变成移动端”的 Flutter 重构项目，而更像一个已经初步成立、正在解决复杂真实问题的移动端产品。

但正因为如此，10.0 也需要比前几版更清醒地意识到一件事：  
当能力越来越多时，真正决定体验上限的，已经不是“再补一个功能”，而是“已有能力之间是不是开始打架、重复、断层、变重”。

所以，10.0 的总体判断不是“当前项目仍然缺很多”，而是：

当前项目已经进入必须做系统收口与行为一致性的阶段。

## 6. 方案 10.0 的核心策略

### 6.1 策略一：从“有多个入口”走向“入口分工清晰”

接下来不应再用“缺什么补什么”的思路继续加入口。  
而应该明确每个入口的职责：

- 顶部索引工具区：解释当前结果状态、承载首次导航动作
- 悬浮索引按钮：承担深滚后的重定位
- 悬浮回顶按钮：承担深滚后的返回起点
- 加词按钮：承担创建行为，不参与导航竞争

只要这层职责划分清楚，当前看似“重复”的入口就会变成“分场景协同”的入口。

### 6.2 策略二：从“已经防抖”走向“搜索状态完全一致”

搜索防抖只是第一步，10.0 的真正目标应当是：

- 输入状态可预测
- 提交状态可预测
- 模式切换时状态可预测
- 词本切换时状态可预测

也就是说，接下来不只是优化性能，还要优化搜索状态的一致性规则。

### 6.3 策略三：从“渐进加载”走向“长列表稳定窗口”

当前分页已足够证明方向正确，10.0 不建议推翻它。  
但应明确下一阶段的目标不是“继续把页大小调大/调小”，而是让列表在长时间使用中仍然稳定轻量。  
这意味着：

- 惰性构建继续保留
- 缓存结果继续保留
- 跳转定位继续保留
- 但要逐步减少“窗口只增不减”的长期累积成本

### 6.4 策略四：让播放页彻底进入“单词对象优先”

当前播放页已经有了更多动作，但仍未真正围绕“当前词本身”建立第一交互层。  
10.0 应把重点从“继续增加控制按钮”转移到“把当前词做成主对象”：

- 当前词先被看到
- 当前词先被理解
- 当前词先可操作

一旦这件事成立，播放页会从“功能更丰富”真正升级为“学习流更自然”。

### 6.5 策略五：把体验验证前移

当前项目的很多问题已经不再是静态代码问题，而是路径问题、节奏问题、状态切换问题。  
这类问题如果没有 widget/integration 级验证，靠人工回归会越来越难压住。

10.0 应明确把测试补齐放入核心计划，而不是放入“有空再做”的末尾工作。

## 7. 分阶段建议

### 阶段 A：先做状态一致性收口

优先处理：

- 搜索防抖状态与搜索模式切换的一致性
- 搜索提交与词本切换的一致性
- 跳转与分页窗口的一致性

这一阶段完成后，用户会明显感受到“页面开始更稳”，而不是只是“看起来多了按钮”。

### 阶段 B：整理导航入口职责

优先处理：

- 顶部索引工具区的角色定义
- 悬浮索引与悬浮回顶的协同规则
- 短结果集场景下的工具降权

这一阶段完成后，词库页会开始具备真正的产品秩序感。

### 阶段 C：继续推进长列表长期稳定性

优先处理：

- 长时滚动后的窗口策略
- 跳转锚点策略进一步减负
- 更完整的惰性构建与软回收思路

这一阶段完成后，大词库场景下的体验才会真正稳定下来。

### 阶段 D：推进播放页对象优先

优先处理：

- 当前词点击即进入跟读/动作层
- 控制区视觉顺序退后
- 当前词状态更强表达

这一阶段完成后，播放页会从“可切词”升级为“可围绕词高效学习”。

### 阶段 E：补齐自动化验证

优先处理：

- 词库页 widget 测试
- 索引弹层路径测试
- 回顶与分页联动测试
- 播放页切词与对象动作路径测试

## 8. 10.0 验收标准

10.0 完成后，建议至少满足以下判断标准：

- 在 1 万词以上词库中，搜索输入与结果刷新保持稳定，不出现明显跳变或迟滞感。
- 用户在快速输入后立即切换搜索模式，不会看到明显的状态错位。
- 深滚状态下，索引和回顶工具都能在 1 步内触达，且角色清晰。
- 词库页长时间滚动后不会持续变重到影响使用节奏。
- 短结果集场景下，工具区会自动降权，不再压过结果本身。
- 播放页中，当前词是第一视觉中心，而不是控制按钮组。
- 用户可以通过当前词主体本身进入跟读或对象动作，而不是必须先寻找按钮。
- 关键体验路径具备自动化验证，不再只依赖人工回归。

## 9. 最终建议

当前项目的状态，其实比 9.0 时更乐观。  
因为它已经不再只是“知道问题在哪里”，而是已经开始把问题一个个翻译成具体实现：缓存做了，防抖做了，Sliver 做了，回顶做了，索引悬浮也做了，播放页切词也做了。

现在真正需要的，不是继续大面积发散，而是把这些已经出现的能力做成一套更稳定的系统。  
因此，10.0 最重要的建议不是“继续加功能”，而是：

把已有能力之间的边界、顺序、状态和职责整理清楚。

如果 9.0 的关键词是“性能托底 + 导航成型”，那么 10.0 的关键词应当是：

让已经成型的能力开始互相配合，而不是彼此抢位置。
