# Vocabulary Sleep Flutter UI/UX 重构反馈方案 12.0

更新时间：2026-03-11

## 1. 版本定位

12.0 的重点，不再是继续泛泛讨论“页面还可以更好看”，而是基于当前最新代码和本轮截图，集中处理两类已经直接影响体验的问题：

- 多语言本地化已经从“是否支持切换”进入“是否真正完整、一致、可信”的阶段。
- 页面适应问题已经从“局部控件有点拥挤”进入“组件级布局策略本身不稳定”的阶段。

本轮和 11.0 的一个重要区别是：当前项目已经补上了应用级 `locale`、`FlutterLocalizations` 以及“跟随系统语言”能力，所以 12.0 不会再把“语言体系完全缺失”作为主结论。  
相反，当前真正需要解决的是：

- 文案覆盖仍明显不完整，导致中文界面中仍出现大片英文回退；
- 一些核心组件在窄宽度下的布局策略不合理，导致 More 页箭头错位、外观页密度失衡、词库索引预览出现“半截元素”的不稳定观感。

因此，12.0 的核心判断是：

当前项目已经从“补语言能力”进入“做语言质量”，从“补移动端入口”进入“做移动端布局系统”。  
如果这一轮不收口，后续页面越多，产品的完成度会越来越依赖运气，而不是依赖稳定的设计系统。

同时结合了你本轮提供的三张真实截图，重点关注以下问题：

- 词库页索引预览中右侧字母被截断，出现“半个 chip”；
- More 页设置卡片的箭头被挤到下一行左下角，导致视觉层级断裂；
- 外观工作室在中文环境下仍出现大量英文标题与标签，同时窄宽度下控件堆叠密度明显失衡。

## 3. 当前版本值得肯定的进展

先明确一点，这一轮并不是“做了很多却没有进步”。  
相反，当前版本在多语言链路上已经有几项明确进展：

### 3.1 应用级 locale 已经接通

`main.dart` 当前已经为 `MaterialApp` 接入：

- `locale`
- `supportedLocales`
- `GlobalMaterialLocalizations`
- `GlobalWidgetsLocalizations`
- `GlobalCupertinoLocalizations`

这说明项目已经不再停留在“只换业务文案，不换组件环境”的阶段。  
这是很重要的正向进步，意味着语言设置已经开始真正影响应用级运行环境。

### 3.2 “跟随系统语言”已经落地

`SettingsService` 当前已经引入 `uiLanguageSystem`，默认值也不再固定中文，而是系统语言。  
`LanguageSettingsPage` 也已经支持“跟随系统 / 手动指定语言”两种模式。

这说明团队已经开始从“手动切换语言”走向“默认就尽量正确”，这是多语言产品从可用走向成熟的关键一步。

### 3.3 系统词本显示名已经开始本地化

`wordbook_localization.dart` 当前已经对 `Favorites / Task` 做运行时本地化映射。  
这说明项目已经不再完全把系统词本名当成数据库原始文本展示，而是开始向“显示名本地化”靠拢。

### 3.4 页面分层已经更清晰

词库、设置中心、More 聚合页、外观工作室已经形成较清晰的信息结构。  
这意味着本轮看到的很多问题，其实不是“信息架构混乱”，而是“同一套信息结构在不同宽度下没有稳定落地”。

这也是 12.0 的重点：不是重做页面结构，而是把现有结构做稳。

## 4. 当前最值得重视的问题

## 4.1 多语言本地化：链路已经建立，但完成度仍然不够

### 4.1.1 当前不是“不能切换语言”，而是“切换后仍明显不完整”

`AppI18n` 当前英文主表约有 `363` 个 key。  
本轮统计到：

- 中文显式覆盖约 `285` 个 key，仍缺 `78` 个；
- 日语、德语、法语、西语显式覆盖约 `279` 个 key，各自仍缺 `84` 个。

这些缺口不是均匀分布的，而是明显集中在外观与体验配置相关区域。  
这正好对应了你截图里“外观工作室中文界面仍大量显示英文”的现象。

换句话说，当前项目的多语言问题已经不是“全局没做”，而是“局部高曝光区域没补齐”，这会比完全没做更影响观感，因为它让用户产生“已经切到中文，但为什么这页还是英文”的割裂感。

### 4.1.2 外观工作室是当前本地化缺口最集中的页面

本轮确认，中文环境下缺失显式翻译的键值大量集中在 `appearance*` 这组文案上，例如：

- `appearanceReset`
- `appearanceTypographyTitle`
- `appearanceFontFamily`
- `appearanceFontScale`
- `appearanceTitleWeight`
- `appearanceBodyWeight`
- `appearanceWeightRegular / Medium / Semibold / Bold`
- 大量背景图、动效、透明度、主题自定义相关键值

这就直接解释了截图里的现象：

- 页面标题是中文；
- 但分区标题仍是 `Typography`；
- 按钮还是 `Reset`；
- 下拉项还是 `System / Semibold / Regular`；
- 中文说明和英文控件标签混排在一起。

这类问题的伤害非常直接，因为它发生在“用户正在调整视觉风格”的场景里。  
当用户进入一个强调产品品质的页面，却看到混杂的语言层级时，页面本身的专业感会被明显削弱。

### 4.1.3 本地化仍是双轨结构，维护成本已经开始明显抬升

当前项目仍同时依赖两条文案路径：

- `AppI18n.t(key)` 的中心化 key 表；
- `pickUiText(...)` 的页面内即时双语文案。

本轮统计到：

- `pickUiText(...)` 总调用约 `402` 处；
- 其中只有 `4` 处真正补齐了 `ja / de / fr / es`；
- `397` 处仍然只是 `zh + en` 双语结构。

这意味着即便应用已经支持 6 种语言，页面级文案的实际维护模型仍然更接近“中英双语产品 + 其他语言局部补丁”。  
短期看这种写法效率高，长期看会带来三个问题：

- 很难知道某页到底该去 `AppI18n` 找文案，还是在 UI 文件里直接改；
- 很多日德法西语页面会持续回退到英文，完成度难以上来；
- 同一语义在不同页面里可能出现不同译法，难以统一。

### 4.1.4 “跟随系统语言”目前更像初始化快照，而不是真正的动态跟随

当前 `AppState` 会通过 `PlatformDispatcher.instance.locale` 解析系统语言，并在初始化或用户主动切换到“跟随系统”时生效。

这已经比旧版更好了，但它仍有一个产品层面的限制：

如果用户在应用运行中途修改系统语言，当前状态并没有看到针对系统 locale 变化的主动监听与自动刷新链路。

这说明当前的“跟随系统”更接近：

- 初始化时跟随一次；
- 手动切回时重新解析一次；

而不是：

- 只要系统语言变了，应用就持续同步。

这不是眼前最显性的 UI bug，但如果未来继续强调多语言，这会成为体验一致性的下一层问题。

### 4.1.5 语言质量问题已经从“技术能力”转向“翻译治理能力”

当前 `main.dart`、`SettingsService`、`LanguageSettingsPage` 这些底层能力已经开始成形。  
因此，12.0 不建议再把大量精力放在“怎么接 Flutter 本地化”这种底层问题上，而应该转向更实际的治理问题：

- 哪些高曝光页面必须先补齐中文显式翻译；
- 哪些业务文案应统一收口进 `AppI18n`；
- 哪些文案仍允许 `pickUiText` 存在；
- 如何在发版前检查“本轮新增页面有没有把非英文环境做完整”。

也就是说，当前项目的本地化问题已经不是“会不会做”，而是“有没有机制保证一直做对”。

## 4.2 页面适应异常：问题已经是组件级，而不是单页级

### 4.2.1 More 页箭头掉到左下角，不是偶发 bug，而是当前组件策略本身如此

你截图里 More 页设置卡片的箭头跑到左下角，这个现象并不是样式抖动，而是 `SettingTile` 的窄宽度布局分支主动这么设计的。

当前逻辑是：

- 当 `constraints.maxWidth < 380` 时，进入 `isNarrow`；
- 图标 + 文本块先占一行；
- `trailing` 再单独放到下一行，并额外 `padding left: 36`。

这在代码层面看起来像“移动端兜底”，但从实际效果看，会产生明显的语义断裂：

- 箭头不再和该行内容构成一个整体；
- 用户的视觉动线从“卡片信息 -> 进入”被打断成“卡片信息 + 底部残留箭头”；
- 尤其在只使用默认 chevron 时，这个箭头会显得像误落下来的孤立图标。

更重要的是，这个分支并不是少数极端宽度才会触发。  
对于常见手机宽度来说，页面左右已经有 `16 + 16` 的外边距，卡片内又有 padding，因此很多正常手机宽度最终都会落入 `< 380` 这一分支。

换句话说，当前的“窄屏模式”在真实移动端上并不是兜底，而几乎是默认状态。  
这也是为什么这个问题会在截图里如此明显。

### 4.2.2 词库页索引预览出现“半截字母”，说明预览层设计和正式导航层职责冲突

当前词库页已经有两层索引能力：

- 顶部卡片里的 `展开索引` 主按钮；
- 下方的字母预览 chip 行。

问题在于，这条预览行使用的是 `SingleChildScrollView + Row + ActionChip`。  
当可见字母较多时，预览行会自然超出卡片宽度，并在初始状态下露出一个“半截 chip”。

这会带来两个体验问题：

- 用户看到的是一个被截断的字母入口，而不是一个稳定完整的导航区域；
- 既然上方已经有“展开索引”这个正式入口，预览层再依赖横向滚动，反而会让导航职责变得重复和不清晰。

也就是说，当前问题不是简单的“chip 太多”，而是“预览层本应做概览，却被做成了另一个需要横向滚动的次级导航带”。

从产品角度看，这种设计最容易出现“没有报错，但观感总觉得不稳”的问题。  
你截图里被圈出的 `F` 正好就是这个现象的典型表现。

### 4.2.3 外观工作室当前是功能齐了，但没有建立 compact width 的真正策略

`AppearanceStudioPage` 当前总体上是单列长表单结构，几乎所有分区都采用：

- 固定 `Card + Padding(18)`
- 固定纵向堆叠
- 固定间距
- 固定表单控件大小

这套结构在中等宽度上是可用的，但在窄宽度场景下会出现三个明显问题：

第一，信息密度不均衡。  
有些分区标题很大，但下面立刻叠加多个下拉框和滑杆，阅读节奏偏重，缺少针对窄宽度的节奏压缩。

第二，英文回退会进一步放大拥挤感。  
当页面顶部还是中文，但中间突然出现 `Typography / Font family / Font scale / Semibold / Regular` 这类英文标签时，视觉层级会一下子被拉散。

第三，这个页面虽然提供了“Compact layout”配置项，但页面本身的编辑界面却没有真正采用 compact-width 编辑策略。  
也就是说，用户在调整“页面是否更紧凑”，自己所处的编辑页却没有先做到紧凑与自适应。

这会造成一种很典型的反差：

- 产品在概念上讨论“布局密度”；
- 但设置页本身却没有根据当前窗口宽度做出密度响应。

### 4.2.4 当前适配问题的源头已经集中到少数基础组件

本轮非常明确的一点是：这些问题已经不是每个页面都要单独修，而是集中来源于少数基础组件和基础策略：

- `SettingTile` 的窄屏 trailing 排布策略；
- `LibraryPage` 索引预览层的横向滚动设计；
- `AppearanceStudioPage` 对 compact width 缺乏专门布局；
- `SectionHeader`、表单区块、下拉框和 slider 组合没有形成统一的窄宽度规范。

这是一件好事，因为它说明问题已经被压缩到“组件层”而不是“页面层”。  
只要组件层策略修正，多个页面会一起变稳。

## 4.3 多语言与适配问题正在互相放大

这一轮最值得警惕的，并不是本地化问题和适配问题各自存在，而是它们已经开始互相放大。

例如外观工作室：

- 如果文案都已经是中文，窄宽度下虽然密度高，但仍能勉强维持统一感；
- 但当页面在窄宽度下同时出现大量英文回退时，英文单词长度、视觉重心和字体识别节奏都会打破原本的阅读流。

再例如 More 页：

- 如果 SettingTile 的 trailing 是完整中文标签，也许还能被理解为“二级说明”；
- 但当前 trailing 是单独一个箭头图标时，一旦掉到底部，就会更像排版残留，而不是有效的交互提示。

因此，12.0 的修复不应该拆成两个彼此独立的任务：

- 一组人补翻译；
- 另一组人改布局。

正确的方式应当是把它看成“高曝光页面的一体化质量收口”。

## 5. 12.0 方案建议

## 5.1 P0：先处理截图里最影响信任感的问题

这一阶段建议优先处理三件事：

### P0-1 外观工作室中文显式翻译补齐

优先补齐 `appearance*` 相关高曝光键值，至少先覆盖：

- 分区标题
- Reset / Theme / Font family / Font scale / Weight 等核心标签
- 背景图与动效区的高频操作项

这一项的目标不是一次性补完所有语言，而是先让中文界面不再出现明显的英文回退。

### P0-2 重做 `SettingTile` 的窄宽度 trailing 策略

More 页截图说明，当前“trailing 掉到下一行”并不适合默认移动端。

建议把窄宽度策略改成以下方向之一：

- 箭头永远保持在首行右侧，subtitle 自己换行；
- 如果 trailing 是图标，则不进入单独换行分支；
- 只有 trailing 是较长文本时，才允许转为第二行，并且要和正文形成明确分组，而不是单独漂浮。

重点不是把箭头移回去，而是重新定义“移动端 SettingTile 在窄宽度下什么信息优先保留在首行”。

### P0-3 词库索引预览层不再允许“半截 chip”作为默认状态

当前上方已经有“展开索引”主按钮，因此字母预览层应降级为概览，不应该再承担完整导航责任。

建议方向：

- 预览层改为 `Wrap`，只展示前 4 到 6 个代表性字母；
- 超出部分统一用“更多”或“全部索引”承接；
- 禁止默认首屏出现半截字母；
- 如果确实保留横向滚动，也应加明确 fade 提示或整块可拖动暗示，而不是自然露出半个元素。

## 5.2 P1：把本地化和适配从修页面，升级成修系统

### P1-1 建立高曝光页面翻译完整度清单

当前最需要优先收口的页面应包括：

- `AppearanceStudioPage`
- `SettingsHomePage`
- `MorePage`
- `LibraryPage`
- `HelpCenterPage`

这些页面是用户高频接触或最容易形成“产品印象”的区域。  
应优先做到：

- 中文完整；
- 英文完整；
- 其他语言即便暂时回退，也尽量回退到统一路径，而不是页面里一半 `AppI18n` 一半 `pickUiText`。

### P1-2 为基础组件建立 width class 规则

当前页面大量问题都源于“窄宽度怎么办”没有统一策略。  
建议以组件为单位建立规范：

- `compact`：手机窄宽度
- `regular`：普通手机 / 小平板
- `wide`：平板 / 桌面

并明确每个组件在不同宽度下：

- 标题和副标题是否同列；
- trailing 是不是必须首行对齐；
- chips 是否允许横向滚动；
- 表单控件的间距、padding、说明文字是否需要压缩。

只要这套规则立住，后续不需要每个页面重复“猜布局”。

### P1-3 逐步减少页面内零散 `pickUiText`

当前 `pickUiText` 仍有约 `402` 处调用，其中绝大多数只覆盖中英。  
12.0 不建议一次性全部迁走，但建议先把下面两类收口进 `AppI18n`：

- 页面标题、副标题、分区标题
- 按钮、设置项、空状态、操作提示

这样后续多语言扩展时，至少最显眼的那一层不会再散落在页面里。

## 5.3 P2：进入可回归、可验证的质量治理

当 P0/P1 处理后，建议进入下一层：

### P2-1 建立本地化覆盖检查

至少应支持：

- 新增英文 key 后，检查中文是否缺失；
- 高曝光分组如 `appearance*`、`settings*` 是否存在空洞；
- 发版前输出“本轮新增 key 缺失情况”。

### P2-2 建立响应式回归检查

当前项目最缺的不是单元测试，而是对不同宽度的 UI 回归保护。

建议至少覆盖以下宽度：

- `320`
- `360`
- `390`
- `412`
- `600`

重点检查：

- More 页卡片 trailing 是否仍保持正确层级；
- Library 页索引预览是否出现半截元素；
- Appearance 页是否出现混语和局部溢出；
- 大字号或系统字体变化后是否仍稳定。

### P2-3 将“语言质量”和“布局质量”纳入同一轮验收

当前项目已经证明，这两类问题会互相放大。  
因此不建议未来继续把它们拆成两张互不关联的验收单。

更合理的方式是：

- 每次新页面上线，同时验本地化完整度和 compact width 表现；
- 每次大规模新增设置项时，同时跑翻译与窄宽度检查；
- 每次高曝光页改版时，优先看“用户看到的是不是一个完整页面”，而不是先看“功能有没有都挂上去”。

## 6. 本轮结论

当前项目已经完成了一件很重要的事：  
多语言底层能力和移动端信息结构都已经搭起来了。

所以现在最需要的，不是再补更多入口，而是把已经搭起来的这套系统做得稳定、完整、可信。

从本轮截图和代码看，当前最核心的问题可以概括为两句话：

第一，语言能力已经有了，但高曝光页面的翻译完成度还不够，尤其外观工作室会直接暴露这种缺口。  
第二，移动端适配能力已经有了，但部分组件在窄宽度下采用了错误的降级策略，导致视觉层级和交互暗示同时失真。

如果把 11.0 的重点概括为“语言、资源、数据底层治理开始成形”，那么 12.0 的重点就是：

“当底层能力已经接上之后，真正决定产品完成度的，是高曝光页面的一致性和组件级适配质量。”

这也是 12.0 最明确的建议：

- 不要再把截图里的问题当成单个页面 bug；
- 要把它们当成组件策略和本地化治理的系统性信号；
- 先修最影响信任感的高曝光页面，再把修法沉淀成可复用规则。

只有这样，后续继续迭代时，产品才不会每做一页就重新冒出一批新的混语和适配问题。
