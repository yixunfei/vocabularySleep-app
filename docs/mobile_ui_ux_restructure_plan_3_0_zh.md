# Vocabulary Sleep Flutter UI/UX 重构方案 3.0

更新时间：2026-03-10

## 1. 版本说明

本文件是在《移动端 UI/UX 重构方案》1.0 与 2.0 的基础上，结合 `flutter_app` 当前最新实现状态重新评估后形成的 3.0 版本方案。3.0 的判断不再建立在“预计将要做什么”上，而是建立在“当前代码已经真实落地了什么、哪些问题已经被解决、哪些问题成为新的主要矛盾”之上。

与 2.0 相比，当前版本已经出现了明显的阶段跃迁。应用入口已经稳定为 `main.dart -> AppShell -> NavigationBar + IndexedStack` 的移动端壳层结构；`MorePage` 中面向普通用户的 legacy 工作台入口已经被移除；`SettingsHomePage` 以及 `PlaybackAdvancedPage`、`VoiceSettingsPage`、`RecognitionSettingsPage`、`AppearanceStudioPage`、`LanguageSettingsPage` 等二级设置页已经补齐；`PracticePage` 也不再只是“当前词功能集合”，而是开始通过 `PracticeSessionPage`、`ReviewSessionPage`、`FollowAlongPage` 形成更接近正式产品的练习链路。

因此，3.0 的目标已经不是“把旧首页拆开”，也不是“继续证明新架构方向是对的”。这一阶段真正要做的，是让当前已经成型的新结构从“架构已成立”进入“体验已成熟”，也就是从过渡型产品原型，进入可以长期演进、可以稳定承接移动端用户心智的正式产品版本。

## 2. 评估依据

本次评估基于当前 `flutter_app` 中已经实现的真实代码结构进行，重点查看了以下路径所代表的核心产品层：应用入口与壳层、一级导航、播放页、练习页、更多页、设置中心、跟读页、主题系统与全局忙碌态。同时，本次还对当前代码质量进行了一次基础验证，`flutter analyze` 与 `flutter test` 在 2026 年 3 月 10 日均已通过，说明当前版本在工程健康度上是可控的，这为后续 3.0 方案从“讨论”转向“执行”提供了比较好的基础。

从评估视角上，3.0 关注的不只是“页面有没有”，而是以下五个层面是否已经具备正式产品特征：第一，用户路径是否统一；第二，页面职责是否清晰；第三，设置与练习是否已经形成独立产品系统；第四，视觉与文案是否足够一致；第五，遗留结构是否已经从主路径中退出。

## 3. 当前版本的总体判断

当前版本可以明确判断为：2.0 阶段提出的多数关键方向已经落地，新架构已经不再是试验品，而是实际承担主产品路径的正式结构。`PlayPage`、`LibraryPage`、`PracticePage`、`MorePage` 组成的四个一级页面已经具备稳定主导航意义；`SettingsHomePage` 说明设置体系已经从单页大控制台式结构进入分层管理；`PracticePage` 及其会话页说明练习功能也已经开始脱离“当前词附属工具”定位，转向“会话式产品模块”。

如果用一句话概括当前阶段，那么就是：项目已经完成了“从旧首页迁移到移动端产品骨架”的关键一跃，但 3.0 仍然必须完成“从结构正确走向体验成熟”的第二次收口。

这意味着当前版本已经不再以“遗留架构妥协”作为主要问题。新的主要矛盾，已经转为：虽然结构完整度显著提升，但页面之间的体验标准、文案系统、场景表达、结果反馈与产品闭环能力还没有完全统一，仍能让用户感受到明显的“重构后期收尾中”痕迹。

## 4. 2.0 阶段已经完成的关键成果

首先，最重要的一点是用户主路径已经真正切换到新结构中。当前 `AppShell` 已经承担应用壳层、底部导航、忙碌态覆盖层与迷你播放器管理职责，这说明项目已经不再依赖旧 `HomePage` 作为默认主舞台。从代码入口关系来看，`home_page.dart` 目前也已经不再出现在主路径引用链中，这意味着 2.0 阶段最核心的架构目标之一实际上已经达成。

其次，设置中心的分层化已经落地，这比 2.0 时期“建议拆页”更进一步。当前 `SettingsHomePage` 已经形成一个清晰的设置枢纽，并把播放、语音、识别、外观、语言等能力拆分到独立页面中。这个变化非常关键，因为它代表应用已经不再把“所有高级配置放进一个大页里”当作默认做法，而是开始按用户认知维度组织设置结构。这种信息架构更符合移动端，也更利于后续做引导、摘要、预览和状态反馈。

第三，`MorePage` 的角色已经明显收敛。它不再把“新页面入口”和“老工作台入口”混在一起，而是更明确地承担设置中心、词本管理、数据管理、关于与帮助等管理性入口分发职责。这一变化让一级导航的心智清晰度提升了很多，也使“更多页”开始接近一个正式产品页，而不再只是重构中的过渡中转站。

第四，练习系统已经从单点功能变成了初步的产品链路。当前 `PracticePage` 已经不是简单列几个按钮，而是具备“当前练习快照 + 多种会话入口 + 会话页 + 复习页 + 跟读页”的组合结构。`PracticeSessionPage` 中的题目推进、进度反馈、结果统计与薄弱词回练逻辑，说明团队实际上已经开始实现练习闭环，而不是仅仅把旧首页上的按钮搬过来。

第五，体验模式的产品语义比以前更明确了。当前主题系统不只是支持切色，而是通过 `Sleep / Focus` 这两个模式影响布局密度、背景强度、面板材质与整体信息密度。这个方向依然非常正确，因为它不是把“主题”理解为纯视觉选项，而是把它提升为“场景化产品默认值”的一部分。这也是当前产品最有潜力、最值得继续深挖的差异化能力。

## 5. 当前版本最值得肯定的设计判断

当前项目最值得肯定的，并不只是页面数量增加了，而是几个产品判断已经走在正确轨道上。

第一，底部四大主导航已经真正建立了页面职责边界。播放页负责持续听词与当前播放体验，词库页负责浏览、筛选与内容管理，练习页负责测试、复习与跟读，更多页负责设置和管理。这样的结构意味着未来新增功能不需要再争抢首页空间，而可以围绕既有页面职责自然扩展。

第二，设置系统已经从“功能堆砌”转向“认知分层”。这是移动端可用性中非常重要的一步。用户不需要理解底层技术模块，也不需要在一个超长页面中来回滚动寻找选项，而是可以通过“播放”“语音”“识别”“外观”“语言”这些更自然的分组完成配置。

第三，练习页已经具备“从入口到结果”的雏形。虽然现在还没有形成长期学习数据闭环，但最起码“选择练习方式 -> 连续答题 -> 结束后反馈 -> 针对薄弱词再练一轮”的基本骨架已经存在，这为 3.0 做学习系统升级提供了非常好的基础。

第四，当前主题层与组件层已经具备形成设计系统的条件。`AppThemeTokens`、`PageHeader`、`SectionHeader`、`SettingTile`、`WordCard`、`BusyOverlay` 等部件说明团队已经在从“页面拼装”过渡到“组件化 UI 体系”。只要 3.0 进一步统一页面样式标准、文案口径和交互反馈节奏，这套系统是可以继续长出来的。

## 6. 当前版本仍然存在的核心问题

虽然当前版本比 2.0 时期成熟很多，但从正式产品视角来看，仍然存在几个必须优先处理的问题。这些问题已经不再属于“大结构没搭好”，而是属于“结构搭好了，但完成度还没有收口到位”。

### 6.1 文案与编码质量成为当前最紧急的 UI 风险

这是 3.0 阶段最需要优先解决的问题。当前新增的多个设置页面中，已经能观察到部分中文字符串出现明显乱码痕迹，例如 `PlaybackAdvancedPage`、`VoiceSettingsPage`、`RecognitionSettingsPage`、`AppearanceStudioPage`、`LanguageSettingsPage` 中存在一批疑似编码损坏后的中文文案。由于同一项目中其他页面中文显示正常，因此这更像是源码层面的文案损坏，而不是终端显示问题。

从产品角度看，这不是一个“小瑕疵”，而是当前版本最接近阻断级别的体验风险。因为设置页本身承担的是“解释系统能力”的角色，一旦中文文案乱码，用户就不仅仅是“观感不好”，而是会直接失去对功能含义的理解能力。更关键的是，这类问题也暴露出当前文案体系依然处于“局部页面直接写 `pickUiText` 文本”的过渡阶段，缺乏统一内容资产治理，这会持续放大后续维护风险。

因此，3.0 必须把“设置页文案编码修复 + 新 UI 文案统一收口”放到最高优先级，而不是将其视为后续顺手修补的细节问题。

### 6.2 设置体系虽然已经成形，但仍停留在“功能可配”而非“体验可理解”

当前设置中心最大的优点是结构已经清楚，但最大的不足也在于此：它已经是一个“好用的设置入口”，却还不是一个“成熟的移动端设置产品”。

问题主要体现在三个方面。第一，二级设置页普遍以卡片加控件的方式直接铺开，功能完整但缺少引导层、摘要层与实时预览层。用户可以改，但不一定知道“为什么改”“改了之后会影响哪里”。第二，多个设置页之间虽然视觉样式一致，但信息密度与表达方式几乎完全同质化，导致“播放高级设置”“外观工作室”“语音设置”在情绪感和使用预期上没有被拉开。第三，绝大部分设置变更都采用即时生效模式，但页面本身没有提供足够明确的状态总结与结果反馈，这会降低用户对改动结果的感知确定性。

换句话说，当前设置系统已经完成了结构重构，但还没有完成“体验升级”。3.0 的方向不应该只是继续往里加开关，而应该让设置变成“有摘要、有解释、有预览、有结果”的产品化配置体验。

### 6.3 练习系统已经有了骨架，但还没有形成长期学习闭环

`PracticePage`、`PracticeSessionPage` 与 `ReviewSessionPage` 的加入，说明当前项目已经从“练习按钮集合”迈向“会话式练习模块”。但从学习产品视角看，它仍然停留在一次性会话层，还没有真正形成长期闭环。

目前缺失的主要能力包括：练习结果的持久化、今日练习目标或日内完成度、历史薄弱词的累计管理、推荐下一步动作、不同练习模式之间的统一成绩视图，以及播放学习与练习复盘之间的联动关系。现在的结果页已经能告诉用户“本轮记住了多少、哪些词薄弱”，但这些结果在用户退出当前页面之后并不会自然沉淀为下一次学习的上下文，这意味着练习仍然更像一个独立工具，而不是可持续使用的学习系统。

因此，3.0 不应该再把练习页理解为“补几个入口”就够了，而应该把它升级为真正的第二核心产品模块，让它和播放页形成“输入-巩固”的双主线关系。

### 6.4 播放主路径结构稳定，但沉浸感与控制深度仍偏轻

从结构判断上，当前播放页已经相当稳妥：页头、词本切换、词卡、进度、播放模式、文本开关和快捷动作都放在了合理位置，适合作为默认首页。但这更像是一套“正确的基础骨架”，还不是最终成熟态。

当前播放体验仍存在两个明显的提升空间。其一，播放器的控制层次偏浅，`MiniPlayer` 已经承担底层持续控制，但还没有形成明确的“展开态”或“播放工具层”，用户缺少对队列、定时结束、环境音组合、节奏方案等能力的自然访问方式。其二，`Sleep / Focus` 模式虽然已经作用于主题参数，但在播放页的行为逻辑上还没有被充分放大，例如不同模式下默认文本显隐、动作优先级、环境音入口权重、信息密度和操作路径并没有形成足够强的差异化体验。

这意味着 3.0 应该开始把播放页从“结构良好的首页”升级为“真正具有场景感和持续使用理由的主页面”。

### 6.5 遗留结构虽然已退出主路径，但工程命名与过渡壳层还没彻底收尾

当前版本一个重要进步是 `HomePage` 已经不再出现在主用户路径中，这说明 legacy 退场目标实际上已经接近完成。但从工程收口角度看，还存在一些明显的过渡痕迹，例如 `settings_page.dart` 已经退化为单纯返回 `SettingsHomePage` 的兼容壳文件，部分主题能力仍通过 `LegacyStyle` 参与驱动，说明项目虽然完成了主路径切换，但还没有完成命名、职责和历史包袱的最后整理。

这类问题不会立刻伤害用户，却会持续增加团队认知成本。3.0 很适合把这些遗留命名和兼容层列入正式清理计划，因为当前已经具备较好的时机，不必再为大规模功能迁移保留过多临时桥接结构。

## 7. 3.0 的总体判断

3.0 阶段的核心判断可以概括为一句话：当前项目已经不再需要“继续拆页”，而需要“统一完成度”。

所谓“统一完成度”，具体包括五层含义。第一，统一用户心智，确保每个页面都代表一个稳定职责，而不是临时承接遗留能力。第二，统一设计标准，让设置页、练习页、播放页和更多页在信息组织与反馈节奏上进入同一成熟度水平。第三，统一文案系统，把所有面向用户的字符串从过渡式混搭管理，收口到更稳定的内容治理体系里。第四，统一产品闭环，让播放、练习、设置和复盘之间形成持续可用的日常使用路径。第五，统一遗留清理节奏，把现在仍存在的兼容壳文件和 legacy 风格依赖纳入明确退场计划。

因此，3.0 不是“再做一轮大规模重构”，而是一轮针对正式产品化的深度收口。其目标不是让页面更多，而是让当前已经存在的页面真正变成熟。

## 8. 方案 3.0 的核心策略

### 8.1 策略一：先修复内容系统，再精修页面视觉

当前最优先的动作不是继续补视觉细节，而是先修复内容系统的稳定性，特别是新增设置页里的中文乱码问题和新 UI 文案分散问题。因为一套页面即使结构再合理，只要用户无法读懂，它就无法构成成熟体验。

3.0 建议先做一轮“内容收口专项”，包括：修复异常编码字符串；梳理所有新页面的可见文案；统一术语，如“播放高级设置 / 识别与练习 / 外观工作室 / 跟读练习”等命名口径；明确哪些文案保留在 `ui_copy.dart`，哪些应迁入 `ARB`；建立新增页面默认只能从正式文案入口接入的约束。只有内容系统先稳定，后续视觉优化与交互微调才不会反复返工。

### 8.2 策略二：把设置系统从“配置入口”升级为“可理解的配置体验”

设置中心在 3.0 的任务已经不是“继续拆分类别”，而是补足理解层和结果层。建议把每个设置二级页都按同一结构升级：先给页面一个简洁的摘要区，告诉用户这个页面会影响什么；再给核心配置项提供简短的作用解释；最后通过预览、示例或状态摘要增强结果感知。

例如，`AppearanceStudioPage` 不应只展示一组开关，而应该增加一个“当前主题预览卡”，让用户在页面内就能感知 `Sleep / Focus`、背景增强、磨砂面板、高对比文本对最终效果的影响。`VoiceSettingsPage` 不应只让用户选提供方和调语速，而应补一个可即时试听的预览区。`PlaybackAdvancedPage` 也不应该只是参数面板，它更适合增加“适合睡眠 / 适合专注 / 适合复习”的策略描述，让非技术用户理解这些参数背后的使用场景。

### 8.3 策略三：把练习页升级为正式的学习闭环中枢

当前练习页已经初具规模，3.0 应该正式把它定义为应用第二核心模块，而不是播放页的附属功能集合。建议将练习系统分为三层：练习入口层、会话执行层、结果沉淀层。

入口层负责给出清晰的练习选择，例如当前词热身、当前范围会话、整本词本会话、任务词复习、收藏词复习和跟读练习。执行层负责统一会话反馈机制，例如进度、提示、答案显隐、评分与继续逻辑。结果沉淀层则是 3.0 新增重点，它需要把“本轮结果”沉淀为“下次还可继续用的学习上下文”，包括今日完成情况、累计薄弱词、建议继续项以及可再次进入的结果入口。

一旦这三层打通，练习页就会从“有用的功能页”变成“用户愿意反复打开的第二主场景”。

### 8.4 策略四：把播放页从稳定主页面升级为场景化主页面

当前播放页已经足够稳定，3.0 不建议推翻重做，而应围绕“更沉浸、更清晰、更能持续使用”三个方向增强。

所谓更沉浸，是让用户在播放过程中更容易进入稳定状态，例如加入更自然的展开播放器、更清晰的播放节奏摘要、更顺手的环境音控制和更容易理解的当前模式信息。所谓更清晰，是让用户知道当前是以什么顺序播放、显示文本是否开启、当前会循环几轮、还有哪些工具可用。所谓更能持续使用，则是要把睡眠场景和专注场景真正差异化，例如在 `Sleep` 模式下强调少打扰与定时结束，在 `Focus` 模式下强调信息回看、范围切换与练习导流。

这会让 `Sleep / Focus` 不再只是主题参数，而真正成为产品默认行为策略。

### 8.5 策略五：正式启动遗留结构的收口与归档

由于当前主用户路径已经完全转到新架构中，3.0 可以开始以更积极的方式处理遗留结构。建议建立一份明确的 legacy 收口清单，包括：`home_page.dart` 的归档定位、`settings_page.dart` 这种兼容壳文件的保留必要性、`LegacyStyle` 在新主题系统中的残留职责，以及任何仍仅为过渡兼容而存在的命名与桥接逻辑。

这里的目标不是急着删除代码，而是先在工程层面明确“什么是正式结构，什么是历史兼容”。只要这个边界不清晰，团队后续做功能时就仍然容易走回头路。

## 9. 重点页面与模块升级建议

### 9.1 AppShell 与全局层

当前 `AppShell` 已经具备作为正式壳层的全部基础条件，3.0 的重点是强化它的“全局体验承载”角色。建议在这一层继续统一全局错误提示、忙碌态、迷你播放器和一级导航切换反馈，并让 `BusyOverlay` 的消息传递机制覆盖更多真实场景，而不是停留在通用的“处理中”。例如导入词本、导出数据、切换模型、识别处理中等，都应该在全局层有明确的状态文案。

另外，一级导航当前虽然已经清晰，但切换后的状态记忆和页面恢复体验也可以继续优化，尤其是播放页与练习页之间的切换应尽量保留上下文，让用户形成“正在做的事情没有丢失”的感受。

### 9.2 播放页

播放页 3.0 建议继续保留当前结构，因为它已经证明方向正确。升级重点在于两个方面：一是增强播放控制层，让 `MiniPlayer` 可以展开为更完整的控制面板，承接队列预览、定时结束、环境音混合、播放方案等能力；二是增强页面中的场景提示，使用户能清楚感知当前处于哪种模式、为什么会看到这些信息密度和工具权重。

如果要让播放页真正承担产品首页角色，那么 3.0 最值得投入的一点，是让它具备“默认打开就很好用”的成熟感，而不是要求用户理解很多参数之后才会好用。

### 9.3 设置中心与各二级设置页

设置中心首页已经具备正式入口意义，3.0 建议它继续做“总览页”而不是重新变回大杂烩页面。首页可以进一步补足每个分区的摘要信息，例如当前语音来源、识别状态、当前模式、界面语言等，让用户不进入子页也能感知主要配置现状。

二级页建议形成三种明确页面模型。第一种是“策略页”，例如播放高级设置，适合强调场景与默认方案；第二种是“工具页”，例如语音与识别设置，适合提供当前状态与即时预览；第三种是“工作室页”，例如外观工作室，适合让用户边看边改、快速比较差异。只要这三类页面被清晰区分，设置体系就会从“结构完整”跃迁到“体验成熟”。

### 9.4 练习页、会话页与结果页

练习页 3.0 最重要的升级方向，是增加“结果沉淀感”和“下一步建议感”。当前会话结束后，虽然能查看正确率和薄弱词，但用户还不知道接下来最推荐做什么，也无法在日后回看这些结果。建议增加一个轻量的练习档案层，例如今日练习概览、最近薄弱词、上次未完成会话、建议继续练习入口等。

会话页本身已经具备不错的简洁度，后续要避免的不是“功能太少”，而是“为了增强结果沉淀而把当前会话搞得过重”。因此更推荐把结果数据外移到练习首页或结果页摘要中，而保持当前题内交互继续简洁直接。

### 9.5 跟读页

`FollowAlongPage` 当前已经通过“普通模式 / 高级模式”区分，解决了 2.0 时期高级参数直接压在主页面上的问题，这一点是很好的。3.0 更适合继续沿着“分层解释”方向推进：普通模式只强调录音、识别与评分结果，高级模式才暴露引擎、评分方法和细分 breakdown。

此外，跟读页的结果表达还可以更产品化，例如增加“本次发音建议”的简短自然语言总结，而不是只展示相似度和差异项列表。这样能更直接服务普通用户，而高级信息仍保留在展开态中供重度用户查看。

### 9.6 更多页与管理页

当前 `MorePage` 的方向已经明显比 2.0 时成熟，它已经开始承担一个简洁的“管理和设置入口板”角色。3.0 主要不需要重构，而是建议继续收敛，确保它不再承载任何“为了兼容旧结构而保留的入口”。

词本管理和数据管理页面在这一阶段的重点也不是重做，而是补足风险说明、任务反馈与操作结果表达，让它们更像正式工具页，而不是工程功能面板。尤其是数据管理中的导入、导出、清理、合并等操作，更适合增加更明确的后果说明与成功反馈。

## 10. 视觉系统与内容系统 3.0

从视觉层判断，当前项目已经具备非常不错的基础条件。`AppThemeTokens` 已经使主题色板、表面层、描边、文字和按钮风格得到统一控制；`Sleep / Focus` 双模式也让产品不再只有“亮色/暗色”这样单薄的视觉切换。3.0 不应推翻这套系统，而应继续把它收敛为更稳定的默认体验。

视觉系统 3.0 的核心原则应该是：减少“可调项炫技感”，增加“默认值成熟感”。也就是说，不再试图让所有视觉能力都暴露给普通用户，而是让用户在大多数场景下直接获得足够成熟的默认体验，只在确有价值的地方保留高收益、低风险的定制项。

内容系统 3.0 的核心原则则是：减少页面内临时文案拼装，增加统一内容治理。当前项目已经证明 `pickUiText` 在重构阶段很高效，但 3.0 应逐步建立更正式的内容资产分层规则，至少要保证以下三件事：一，所有高频页面文案来源可追踪；二，所有术语在中英文之间具有稳定映射；三，新增页面不会再引入编码损坏与口径漂移问题。

## 11. 3.0 分阶段实施建议

### 阶段 A：内容收口与阻断问题修复

这个阶段的目标非常明确，就是先消除当前最影响完成度的风险。具体包括修复新增设置页中的异常中文文案、统一设置相关命名、梳理新页面对 `ui_copy.dart` 与 `ARB` 的依赖边界，以及建立新增文案的编码与校对检查流程。只有这一阶段完成，3.0 后续所有视觉和交互优化才有意义。

### 阶段 B：设置系统产品化升级

这个阶段不再新增大批设置分类，而是对现有设置页做体验升级。重点包括增加摘要区、说明区、即时预览区与配置结果感知，让每个设置页都从“表单页”升级为“可理解的配置页”。这是 3.0 中最能直接提升成熟度观感的一步。

### 阶段 C：练习闭环建设

这个阶段的重点是把当前会话结果真正沉淀下来。建议增加今日练习概览、最近薄弱词、建议继续练习入口、上次会话状态和轻量历史结果展示，让练习页成为用户每次回来都能继续接上的模块，而不是每次都从零开始的临时工具。

### 阶段 D：播放场景升级

这个阶段主要围绕播放页和迷你播放器增强展开。建议增加展开态播放器、队列与定时工具、环境音控制归位，以及 `Sleep / Focus` 在播放交互层的差异化表达，让首页真正具备“场景即产品”的感觉。

### 阶段 E：遗留结构归档与全局精修

最后一个阶段才适合处理 legacy 收尾与全局精修，包括兼容壳文件清理、历史命名统一、主题遗留依赖审视、空状态与提示语精修，以及各页面反馈细节收口。这样做的好处是不会在主路径仍频繁变化时过早整理表层代码，而能在结构基本稳定后一次性完成收口。

## 12. 3.0 验收标准

从产品层看，3.0 完成后应满足以下判断：普通用户能够完全依赖新架构完成主要配置、播放与练习流程；练习系统不再只是临时会话，而具备连续使用价值；播放页不再只是“可用首页”，而是具有明确场景特征和长期打开理由；设置页不再只是“能改参数”，而是“能理解配置影响”。

从体验层看，3.0 完成后应满足以下判断：用户在中文环境下不再见到异常文案或编码损坏；核心页面的术语、语气和提示风格一致；设置、练习、播放三条主路径都拥有稳定的空状态、错误态、忙碌态与完成态反馈；`Sleep / Focus` 在视觉和行为层都能够被用户明显感知。

从工程层看，3.0 完成后应满足以下判断：legacy 结构具有明确归档边界；新增 UI 文案接入路径明确且可维护；`flutter analyze` 与 `flutter test` 持续通过；新功能不再回流到旧页面或临时兼容壳层中。

## 13. 最终建议

当前项目已经明显跨过了“要不要重构 UI 架构”这个最难的问题，事实证明这次 Flutter 移动端重构方向是成立的，而且已经进入可收获阶段。现在最不该做的，是再次发散到一轮新的大而全重设计；现在最该做的，是围绕当前已经成型的结构，把完成度、语言质量、设置可理解性、练习闭环和播放场景感逐一收口。

如果 2.0 的关键词是“让新架构成立”，那么 3.0 的关键词应该是：“让新架构真正像一个成熟产品”。

这会是当前阶段投入产出比最高，也最符合项目现状的一步。
