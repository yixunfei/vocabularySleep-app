# Vocabulary Sleep Flutter UI/UX 重构反馈方案 11.0

更新时间：2026-03-11

## 1. 版本定位

11.0 的重点不再是单一页面的视觉微调，而是重新审视当前项目在三条底层能力上的成熟度：多语言本地化、资源生命周期管理、重置与恢复数据能力。

从当前实现看，项目已经不缺“入口级功能”了。语言设置页、数据管理页、离线包管理、外观配置、背景图、词本导入迁移等能力都已经存在。但这也意味着产品已经从“有没有功能”进入“这些功能是否一致、是否可清理、是否可恢复、是否能长期维护”的阶段。

11.0 的核心判断是：当前版本已经具备继续扩展的产品基础，但语言、资源、数据三条链路仍停留在“局部可用”的状态，距离“系统级闭环”还有明显差距。如果这一层不收口，后续越做越多时，体验会越来越碎，维护成本也会越来越高。

## 3. 当前版本的积极进展

先明确一点，当前项目并不是“基础能力缺失”。相反，这一版已经有不少值得肯定的进展：

- 已经有独立的语言设置页，且 `AppState` 能持久化 `uiLanguage`。
- 已经有集中式 `AppI18n`，核心系统文案不再完全散落在各页面。
- 已经有数据管理页，导入、迁移、导出、清空任务词本这些基础动作都具备。
- 已经有安全备份机制，导入词本、迁移旧库、合并词本前会尝试生成备份文件。
- 已经有离线 ASR 模型包与评分包的安装/删除入口，说明团队已经开始关注磁盘资源治理。
- 已经有外观重置入口，说明“配置可回退”这件事已经被纳入产品思路。

这些能力说明项目已经进入“系统化管理”阶段，11.0 需要做的不是从零补课，而是把这些零散能力收束成稳定、统一、可预期的用户体验。

## 4. 多语言本地化复评

### 4.1 当前状态的核心判断

当前项目已经具备多语言能力，但还不是一套真正闭环的本地化系统，而更接近“手工可切换的双轨文案系统”。

一条轨道是 `AppI18n`：集中式 key-value 翻译表，已经覆盖大量核心文案。  
另一条轨道是 `pickUiText`：页面内部直接传 `zh/en`，必要时再补 `ja/de/fr/es`。

这套方案短期可用，但当页面越来越多时，语言一致性会迅速下降。

### 4.2 当前最值得重视的问题

#### 4.2.1 默认语言仍然固定为中文，而不是跟随设备语言

`SettingsService.loadUiLanguage()` 当前默认返回 `zh`。这意味着首次启动时，应用不会参考系统 Locale，而是直接进入中文界面。

这对中文用户不是问题，但对英文、日文、法文、西语用户来说，会直接造成首屏认知成本，也会让“多语言已支持”这件事失去一半价值。

对于一个已经提供 6 种界面语言的产品，默认语言仍固定为中文，说明当前多语言还停留在“可切换”，没有进入“默认就正确”的阶段。

#### 4.2.2 当前本地化链路是双轨的，维护成本已经开始显著上升

`app_i18n.dart` 当前体量已经很大，文件超过 2000 行；与此同时，UI 层还有大量 `pickUiText(...)` 的页面内文案。

本轮统计到：

- `pickUiText(...)` 调用约 390 处
- 其中仅 3 处真正补齐了 `ja/de/fr/es`
- 其余绝大多数仍然是 `zh + en` 双语结构

这意味着虽然设置页允许切换为日语、德语、法语、西语，但很多页面级标题、说明、副标题、提示文案在这些语言下仍然会直接回退到英文。

这会带来一种很典型的产品感受：  
“这个应用表面支持多语言，但切换后很多地方还是半英半本地语。”

这类问题不会立刻导致报错，却会持续损伤产品完成度。

#### 4.2.3 应用语言切换没有真正接管 Flutter 的系统本地化链路

当前 `MaterialApp` 只设置了 `title / theme / home`，没有看到与应用语言联动的：

- `locale`
- `supportedLocales`
- `localizationsDelegates`

这意味着当前的“语言切换”主要只覆盖自定义业务文案，而不是完整接管 Flutter 组件层本地化。

直接后果是：

- 对话框里的系统按钮文案依赖 `MaterialLocalizations`
- 各类组件的内建文案与交互顺序，未必和应用内选中的语言保持一致
- 应用选择了某种语言，不等于整个界面都进入该语言环境

也就是说，当前语言设置更像“业务层换肤”，还不是严格意义上的“应用级 Locale 切换”。

#### 4.2.4 系统词本名称仍是持久化文本，不是真正的本地化显示名

`database_service.dart` 中系统词本以 `"Favorites"`、`"Task"` 作为数据库持久化名称写入；  
`wordbook_management_page.dart` 展示时直接显示 `book.name`。

这会带来两个问题：

- 中文界面下，系统词本仍可能显示英文名称
- 语言切换后，系统词本名称不会跟着切换

这说明“界面语言”与“系统数据展示名称”目前还是分离的。用户看到的是一个中文界面里夹着英文系统词本名，这会影响整体统一性。

#### 4.2.5 语言设置目前只覆盖 UI 语言，没有形成“语言偏好中心”

当前 `LanguageSettingsPage` 只有一个“界面语言”下拉项，且是即时切换。

但从实际使用看，项目里至少还有三类与语言相关的设置没有被放进同一条认知链路：

- TTS 的语言/声音偏好
- ASR 的识别语言
- 词本内容语言或双语方向

例如当前 `PlayConfig.defaults.asr.language` 默认是 `en`，这并不会随 UI 语言或词本类型自动调整。  
这意味着用户把界面切到中文，不代表识别语言也变得合理；导入中日、中法词库时，也没有默认联动策略。

从产品角度说，现在的“语言设置”更像单点功能，而不是完整的语言体验治理。

#### 4.2.6 帮助中心与产品文案存在版本陈旧问题

`help_center_page.dart` 中仍然保留“4.0 阶段”“当前版本 4.0”之类的内容。

这不是简单的文字遗留，而是本地化内容治理问题：  
当帮助文案、页面说明、版本信息不跟随当前产品阶段时，用户会直接感受到“内容不可信”。

在多语言产品里，过期文案的伤害比普通文案错误更大，因为用户很难判断到底是翻译没更新，还是产品本身没更新。

### 4.3 11.0 对多语言的建议方向

11.0 不建议继续扩大 `pickUiText` 的使用范围，而应尽快收口：

- 把高频页面标题、副标题、空状态、帮助文案逐步收回到统一 key 管理
- 将“跟随系统”作为语言设置默认项，而不是强制固定中文
- 让应用级 `locale` 与业务语言状态绑定，避免组件层和业务层语言脱节
- 系统词本名使用“稳定 ID + 运行时本地化显示名”，不要把显示名直接写死到数据库里
- 语言设置页升级为“语言与区域偏好”，至少要纳入 UI、识别语言、语音语言之间的关系说明
- 为多语言切换建立专项测试，不再只依赖肉眼抽查

## 5. 资源清理与资源生命周期复评

### 5.1 当前状态的核心判断

当前项目已经开始管理资源，但还没有形成“资源从引入、使用、缓存、删除到恢复”的完整生命周期。

目前更像是“部分资源已经有入口和删除按钮”，但资源治理仍分散在多个页面和服务里，缺少统一视角。

### 5.2 当前最值得重视的问题

#### 5.2.1 `assets/wordbooks/` 很可能已经成为历史遗留资源

本轮检查中看到：

- `dict/` 目录约 100.39 MB，并且当前 `database_service.dart` 会基于 `dict/` 自动播种内置词本
- `assets/wordbooks/` 目录约 34.19 MB，`pubspec.yaml` 仍然声明了它，但代码中没有发现明确使用链路

这类资源最危险的地方在于：  
它不会直接报错，也不会立刻影响功能，但会悄悄抬高包体、构建时间和后续维护成本。

如果 `assets/wordbooks/` 已经是旧方案遗留，那么它现在更像“沉默的包体税”。

#### 5.2.2 仍存在体量较大的历史 UI 文件未参与当前主流程

`lib/src/ui/home_page.dart` 当前体量较大，约 279 KB，但主入口已经切换到 `AppShell + Play/Library/Practice/More` 结构。

这说明项目内部仍保留了一份相对完整的旧 UI 逻辑。  
这种遗留文件短期看似无害，长期会带来三类问题：

- 新人难以判断真实入口与历史入口
- 重构时容易改错文件
- 文案、交互、主题逻辑容易出现双份维护

对当前阶段来说，这已经属于典型的结构性资源负担。

#### 5.2.3 背景图资源是“外部路径引用”，不是“应用内受管资源”

当前背景图由 `pickBackgroundImageByPicker()` 选择文件路径后，直接把路径写进配置；  
`AppBackground` 渲染时再检查该路径是否仍存在。

这种方式实现成本低，但资源生命周期并不稳定：

- 文件一旦被用户移动、删除、重命名，背景图会直接失效
- 应用没有把资源导入到自己的受控目录
- 数据管理页也没有“清理失效背景资源”或“查看当前背景资源占用”的入口

这说明当前图片资源管理还不是“资源导入”，而是“路径借用”。

#### 5.2.4 离线模型、评分包、背景图、备份文件、环境音文件，资源治理视角仍然分裂

项目里已经出现多种不同类型的资源：

- 内置词库资源
- 安全备份数据库
- 离线 ASR 模型
- 发音评分包
- 背景图片
- 外部环境音文件

但这些资源分别散落在：

- 识别设置页
- 外观工作室
- 数据管理页
- Ambient 相关交互

用户当前无法在一个地方回答这些关键问题：

- 我的应用一共占了多少空间？
- 哪些资源是系统内置的，哪些是我后续下载的？
- 哪些资源可以删，删了会影响什么？
- 哪些资源已经失效但还留着配置引用？

这说明当前资源管理还是功能视角，不是用户视角。

#### 5.2.5 备份已有创建能力，但缺少清理策略与恢复链路

当前导入、迁移、合并前会创建 safety backup，这是一件正确的事。  
但从产品闭环看，还缺两块关键能力：

- 备份可恢复
- 备份可清理

现在数据管理页只显示 `lastBackupPath`，用户知道“有一个备份”，但不知道：

- 备份列表在哪里
- 最近有几份
- 哪一份对应哪次操作
- 是否可以一键恢复
- 是否需要清理旧备份

这会让安全备份停留在“工程保险丝”层面，而没有升级为“用户可感知的恢复能力”。

### 5.3 11.0 对资源治理的建议方向

11.0 建议把“资源清理”从零散按钮，升级成统一策略：

- 明确区分内置资源、下载资源、用户引用资源、备份资源
- 为大资源建立统一的“占用统计 + 来源说明 + 删除后果提示”
- 清理已废弃资源目录与旧 UI 文件，避免继续隐性增肥
- 背景图、外部音频这类用户资源，应优先导入到应用可控目录，而不是长期保存原始外部路径
- 对备份文件建立保留策略，例如最近 N 份、超期清理、按操作类型归类

## 6. 重置数据与恢复能力复评

### 6.1 当前状态的核心判断

当前项目已经具备“局部清理”能力，但还没有形成“按影响范围分层重置”的产品设计。

这意味着用户能做一些清理动作，但无法明确、安全地完成真正需要的重置任务。

### 6.2 当前最值得重视的问题

#### 6.2.1 数据管理页目前仍偏“导入导出工具页”，不是“数据治理中心”

当前 `DataManagementPage` 提供的是：

- 导入词本
- 迁移旧数据库
- 导出任务词本
- 清空任务词本
- 显示最近备份路径

这些功能都合理，但仍偏向“数据搬运操作”。  
用户实际更关心的很多重置动作并没有入口，例如：

- 重置练习统计
- 清空收藏词本
- 清空最近薄弱词
- 恢复默认播放与识别配置
- 恢复默认外观配置与背景图
- 清理下载的离线资源
- 恢复到某个安全备份
- 全量恢复出厂状态

因此当前页面更像“工具集合”，还不是“数据控制台”。

#### 6.2.2 现在没有一套清晰的分层重置模型

对用户来说，“重置数据”并不是一个动作，而至少应该分成几层：

- 仅重置界面与外观
- 仅重置播放/识别配置
- 仅清空练习记录
- 仅清空任务/收藏/自定义词本
- 清理下载资源但保留词库
- 全量恢复出厂

当前项目里已有的“Reset”主要集中在外观页，而且只是把 `appearance` 恢复为默认值。  
这说明产品已经意识到“可回退”重要，但还没有把这个思路扩展到整个应用状态。

#### 6.2.3 备份链路是单向的，只能创建，不能回滚

当前导入、迁移、合并前都可能生成安全备份，这很好。  
但如果用户之后发现数据错了、词本混了、误删了，当前 UI 并没有“从备份恢复”的路径。

这会导致一个很常见的真实问题：

产品内部其实已经做了防护，但用户在感知上仍然像“没有撤销能力”。

也就是说，当前备份机制在工程上存在，在体验上却仍然接近不可见。

#### 6.2.4 持久化状态存在重复记录，后续重置时容易出现语义不清

当前 `AppState._syncSpecialWordbooks()` 会把 `favorites` 和 `taskWords` 同时写入 `settings`；  
但本轮没有看到对应的读取链路。

这说明这两份数据更像是历史遗留或冗余快照，而不是当前可靠的单一数据源。

在没有统一重置模型的情况下，这类“写了但不再读”的持久化数据非常危险，因为后续一旦做“清理设置”“恢复默认”“迁移旧数据”，就会出现这些问题：

- 到底哪份数据才算真实状态？
- 清掉 settings 算不算清掉收藏/任务？
- 恢复后如果 settings 里残留快照，会不会和词本内容不一致？

这类问题平时不容易暴露，但一旦进入重置/恢复场景，会变成典型的隐性 bug 源头。

#### 6.2.5 当前危险操作的“影响说明”仍然偏弱

例如清空任务词本，当前已经有确认弹窗，这是正确方向。  
但从数据治理角度看，还可以更进一步：

- 会影响多少条词？
- 是否可撤销？
- 是否已有最近备份？
- 清空后哪些页面会发生变化？

对重置型操作来说，用户最需要的不是一个“确定/取消”按钮，而是对后果的准确预期。

### 6.3 11.0 对重置数据的建议方向

11.0 建议把“重置”从散落动作升级成分层体系：

- 建立“轻重分级”的重置结构，而不是继续增加单个清空按钮
- 数据管理页升级为“备份 / 清理 / 重置 / 恢复”四分区
- 每个重置动作都明确影响范围、是否可撤销、是否建议先备份
- 备份从“被动生成”升级为“可浏览、可恢复、可清理”
- 清理冗余持久化键值，先统一状态源，再谈重置逻辑

## 7. 方案 11.0 的执行建议

### 7.1 P0：先把底层规则立住

这一阶段建议优先完成：

- 首次启动默认语言改为“跟随系统”，仅当用户主动选择后再持久化固定语言
- `MaterialApp` 绑定应用级 locale，让组件层与业务层语言一致
- 系统词本名改为“ID 固定、显示名本地化”
- 清查 `pickUiText` 高使用页面，把高频页头、帮助文案、空状态统一回收进 `AppI18n`
- 数据管理页补出“重置范围说明”和“备份恢复占位入口”
- 梳理并删除未使用的资源目录与明显不再参与主流程的历史页面

这一阶段的目标不是做完所有事，而是先把未来不会继续恶化的结构规则立住。

### 7.2 P1：把资源与重置真正做成产品能力

这一阶段建议推进：

- 资源使用总览页：统计词库、离线包、背景图、备份、外部音频
- 备份列表页：按时间、操作类型展示，并支持恢复
- 分层重置页：设置重置、练习数据重置、资源清理、全量恢复
- 背景图与外部文件导入到应用私有目录，并提供失效检查
- 补齐多语言完整度校验和语言切换 smoke test

这一阶段完成后，项目才真正具备“可维护的个人数据应用”特征。

### 7.3 P2：进入长期治理

当 P0/P1 稳定后，再继续做：

- 将 `AppI18n` 拆分成更易维护的语言资源结构
- 建立翻译缺失检测、废弃 key 检测、未引用资源检测
- 为帮助中心建立版本同步机制，避免再次出现“界面版本与产品版本脱节”
- 为备份与缓存建立自动清理策略

## 8. 测试与验证建议

当前测试已经覆盖了一部分搜索逻辑、发音评分、设置持久化和基础 UI smoke，这是一个好基础。  
但围绕 11.0 主题，仍明显缺少以下验证：

- 语言切换后关键页面标题/副标题/弹窗按钮是否同步更新
- 首次启动是否正确跟随系统语言
- 系统词本显示名是否能随语言变化
- 清空任务词本、删除词本、迁移旧库后，备份与恢复链路是否正确
- 冗余持久化清理后，收藏/任务状态是否仍一致
- 背景图路径失效后，界面是否有合理兜底
- 离线资源删除后，配置是否自动回退到安全状态

11.0 建议至少把“语言切换”“重置数据”“备份恢复”纳入专项回归范围，因为这三类问题一旦出错，用户感知会比普通 UI 小问题强得多。

## 9. 最终结论

当前版本已经不再是一个单纯的 UI 重绘项目，而是开始进入“个人学习数据产品”的中后段建设阶段。  
这个阶段最重要的，不是再多做几个入口，而是确保语言一致、资源可管、数据可恢复。

如果把 10.0 的重点概括为“功能正在形成系统”，那么 11.0 的重点就是：

“系统已经出现，但底层治理还没有完全跟上。”

其中最需要优先处理的并不是最显眼的视觉问题，而是下面三件事：

- 让语言真正成为应用级能力，而不是页面级拼接
- 让资源真正进入可统计、可清理、可迁移的生命周期
- 让重置与恢复真正成为用户可理解、可操作、可预期的产品能力

这三件事一旦收口，后续无论继续做 UI、做练习闭环，还是做更多词库与离线能力，整个项目都会更稳，也更像一个成熟产品。
