# 工程结构说明

本文档说明当前 Flutter 工程的目录职责与推荐维护边界，方便后续继续组件化整理时保持一致。

## 根目录

| 路径 | 作用 |
| --- | --- |
| `lib/` | 应用源码入口与业务实现 |
| `assets/` | 图标、品牌资源、环境音、内置词书等静态资源 |
| `dict/` | 项目内置词典与词表数据 |
| `scripts/` | 开发、日志、打包辅助脚本 |
| `test/` | 自动化测试 |
| `docs/` | 评估、重构、构建和迁移文档 |

## `lib/` 目录分层

### `lib/main.dart`

最小入口文件，只负责调用 `runVocabularySleepApp()`，避免入口继续膨胀。

### `lib/src/app/`

应用级装配层，负责“怎么启动”和“怎么组装”。

当前文件职责如下：

- `app_bootstrap.dart`: 框架错误捕获、日志初始化、应用启动入口
- `app_dependencies.dart`: 长生命周期服务实例化与 Provider 装配
- `app_root.dart`: `MaterialApp` 根组件、主题与本地化绑定
- `app_identity.dart`: 应用品牌信息、包名与 logo 资源常量

### `lib/src/services/`

面向基础能力或平台能力的服务层，例如：

- 数据库与导入
- TTS / ASR / 语音播放
- 提醒、环境音、专注服务
- 日志与系统交互

建议：

- 服务层尽量不要依赖 UI 组件
- 同一个服务只处理一种核心能力
- 平台相关逻辑优先收口在这里，而不是散落到页面中

### `lib/src/state/`

应用状态协调层，目前以 `AppState` 为核心，负责：

- 汇总服务调用
- 驱动页面状态与业务流程
- 管理本地缓存、筛选、播放、提醒、练习等跨页面状态

建议：

- 复杂业务状态优先抽成私有辅助方法
- 与 UI 强相关但可复用的状态转换可以继续拆成独立 helper
- 不在这里混入平台构建或资源路径常量

### `lib/src/ui/`

界面层，负责页面、组件、布局和主题。

常见子目录职责：

- `pages/`: 页面级组件
- `widgets/`: 可复用组件
- `theme/`: 主题、颜色、字号等视觉定义
- `layout/`: 宽度断点、响应式辅助

建议：

- 页面负责组织，不承担底层平台逻辑
- 通用样式优先沉淀到组件或主题层
- 头部、卡片、状态条等重复区域尽量组件化

### 其他目录

- `lib/src/models/`: 数据模型与配置对象
- `lib/src/utils/`: 纯工具函数、枚举映射、格式化与轻量辅助
- `lib/src/i18n/`: 本地化文案与语言配置

## 当前推荐依赖方向

建议保持以下依赖方向，减少循环耦合：

`app -> services/state -> ui`

更具体一点：

- `app/` 可以依赖 `services/`、`state/`、`ui/`
- `state/` 可以依赖 `services/`、`models/`、`utils/`
- `ui/` 可以依赖 `state/`、`models/`、`utils/`
- `services/` 尽量不要反向依赖 `ui/`

## 本轮整理原则

本轮结构优化刻意保持低风险，只做以下事情：

- 把入口、依赖装配和根组件从 `main.dart` 拆出
- 把品牌常量集中到 `app_identity.dart`
- 增加通用 logo 组件，避免页面头部重复拼装

未做的事情：

- 没有重写业务逻辑
- 没有大规模改动 `AppState`
- 没有重新设计页面状态流转

这样做的目的是先把工程外壳整理清楚，再逐步进入更深层的业务重构。
