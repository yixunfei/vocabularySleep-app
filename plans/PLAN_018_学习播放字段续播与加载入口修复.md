# 计划 018: 学习播放字段续播与加载入口修复

## 基本信息
- **创建日期**: 2026-04-13
- **状态**: 已完成
- **负责人**: Codex

## 目标
1. 深入排查学习播放中“单词播完后不继续播放释义等已配置字段”的根因，并修复播放队列构建与实际播放链路中的问题。
2. 将大词本/按需加载场景中的“加载并播放”入口调整为“仅加载”，让用户在加载完成后显式点击开始播放。
3. 补充最小必要回归测试，覆盖历史配置残留、学习播放字段续播，以及加载后再播放的状态路径。

## 详细步骤
1. 复核 `PlayConfig`、`WordEntry.playbackFields` 与 `PlaybackService`，确认字段为何未进入播放队列。
2. 修正字段启用判定优先级，确保当前重复次数配置优先于旧版残留开关，不再只播 `word`。
3. 梳理播放页与状态层的按需加载入口，拆分“加载词本”和“开始播放”两个动作，避免加载即开播。
4. 统一播放状态语义，避免“已加载待播放”和“播放中暂停”混用同一状态导致 UI 误导。
5. 补充并运行相关测试，覆盖播放字段、状态切换与 UI/假状态对象的兼容性。
6. 更新 `changelogs/CHANGELOG.md` 与本计划执行结果。

## 风险评估
- **风险 1**: 直接调整字段启用优先级，可能影响旧配置里确实希望禁用的字段。
- **缓解措施**: 以“显式重复次数 > 旧布尔开关”为准，只修正播放设置面板当前已启用的字段；补充测试锁定该语义。
- **风险 2**: 拆分“加载”和“播放”后，现有播放页、迷你播放器和测试桩可能出现状态不一致。
- **缓解措施**: 增加独立的“已加载待播放”状态标识，并同步修正 UI 与测试替身。
- **风险 3**: 现有工作区已有播放相关未提交改动，贸然大改可能扩大冲突面。
- **缓解措施**: 仅在 `play_config`、`app_state_playback`、播放页/迷你播放器和对应测试中做最小改动，不触碰 TTS 业务细节。

## 执行结果
- 已确认学习播放“只播单词本体”的直接根因是 `fieldSettings.enabled = false` 的历史残留配置会覆盖当前重复次数设置，导致 `meaning` 等字段在构建播放队列时被误判为禁用。
- 已修正字段启用优先级：当字段重复次数大于 `0` 时，优先进入播放队列，不再被旧版布尔开关拦截。
- 已将大词本延迟加载入口从“加载并播放”改为“首次点击仅加载，加载完成后由用户再次点击开始播放”，避免加载即自动开播。
- 已补强 `PlaybackService` prepared session 的状态清理与解析词条快照，防止停止后遗留旧会话，或预加载回调继续拿到轻量词条。
- 已补充并通过回归测试，覆盖播放字段续播、历史禁用标记兼容，以及大词本首次点击仅加载/二次点击再播放的路径。
