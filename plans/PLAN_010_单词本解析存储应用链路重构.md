# 计划 010: 单词本解析-存储-应用链路重构

## 基本信息
- **创建日期**: 2026-04-10
- **状态**: 计划中
- **负责人**: Codex

## 背景
当前项目已经完成一批单词本文件的基础规整，并通过 `wordbook.v1` 标准的结构验收，可以作为后续系统重构的输入基线。

但现有应用链路仍然存在明显断层：
- JSON 解析层仍保留大量 legacy 别名和宽松猜测逻辑；
- 数据库存储层仍混合承担摘要缓存、兼容字段和原始内容兜底职责；
- 应用模型与 UI 展示层仍围绕旧字段语义设计，尚未以标准结构为主干；
- 一旦继续沿用旧链路导入新的标准词本，仍可能再次引入字段压平、字段吞并和职责漂移。

因此需要启动一项从“标准词本解析”到“数据库存储”再到“应用各端消费”的渐进式重构计划，确保标准化成果真正进入主系统，而不是停留在文件层。

## 目标

### 主要目标
1. 建立以 `wordbook.v1` 为唯一目标结构的标准解析 DTO 和校验链路。
2. 重构导入与落库逻辑，使标准词条可以无损入库，并保留可回放的结构化源数据。
3. 调整仓库层、应用模型和 UI 消费方式，让核心学习、播放、展示与检索模块以标准字段为主。
4. 在不一次性推翻旧系统的前提下，保留必要兼容层并提供迁移、审计与回滚手段。

### 成功标准
- 所有新导入词本必须先转换并通过 `wordbook.v1` 校验器，再进入数据库。
- 数据库存储中可明确区分“检索摘要字段”“标准结构字段”“原始标准 payload”三类职责。
- 详情页可以完整渲染标准字段分组，卡片页仅展示受控摘要，不再依赖历史压平字段兜底。
- 核心链路具备迁移验证能力：至少支持词本导入对比、数据库迁移校验和 UI 回归检查。
- 旧字段兼容层可被标记并逐步收敛，避免继续扩大 legacy 映射范围。

## 详细步骤

### 阶段一：标准解析基线
1. 梳理当前导入器、模型、数据库和 UI 中所有 legacy 字段入口，形成字段职责清单。
2. 新增标准 DTO：
   - `WordbookSchemaV1`
   - `WordbookBookMetaV1`
   - `WordbookEntryV1`
3. 新增结构校验器与校验结果模型，至少覆盖：
   - 顶层结构校验
   - 必填字段校验
   - 字段类型校验
   - 未识别字段审计
   - 潜在语义冲突告警
4. 为标准化词本增加本地验证脚本或测试入口，确保后续批量词本可重复校验。

### 阶段二：导入与数据库重构
1. 重写导入主流程，使其改为：
   - 文件读取
   - `wordbook.v1` 解析
   - DTO 校验
   - 标准字段映射
   - 数据库写入
2. 调整数据库职责边界：
   - `wordbooks` 补充 `schema_version` 与 `metadata_json`
   - `words` 补充 `entry_uid`、`primary_gloss`、`source_payload_json`、`sort_index`
   - `word_fields` 继续承载展示字段，但 field key 仅允许标准字段集合或明确扩展命名空间
3. 保留必要兼容列，但停止把 `content/rawContent/entry_json` 作为主数据源。
4. 提供数据库迁移脚本、幂等校验和最小回滚策略。

### 阶段三：仓库层与应用模型收敛
1. 调整 `WordEntry`、Repository、Service 层读取逻辑，使标准 DTO 成为主来源。
2. 为搜索、收藏、任务词、学习记录等模块补齐稳定关联键，优先保证：
   - `wordbook_id`
   - `entry_uid`
   - `lemma`
   - `primary_gloss`
3. 通过兼容 getter 或适配层维持旧调用点可运行，避免一次性全量爆改。
4. 输出一份模块影响清单，明确哪些页面和功能仍依赖 legacy 字段。

### 阶段四：UI 与学习链路改造
1. 按移动端优先原则重做词条详情字段分组：
   - 核心：释义、音标、词性
   - 用法：例句、搭配、易混淆点、用法说明
   - 语言学：词源、词根、词缀、形态
   - 记忆：记忆法、文化背景、故事
2. 保持卡片页只展示摘要，明确其为预览层，不再假设字段被压平成固定几项文本。
3. 校正播放、学习、词条切换等核心模块对旧字段的硬编码依赖。
4. 为移动端长内容场景提供折叠、子页或分组展示，避免一屏内冲突和横向溢出。

### 阶段五：迁移验证与切换
1. 选取至少 3 类词本进行迁移验证：
   - 中文到目标语言
   - 目标语言到英语
   - 字段较丰富的复杂词本
2. 对比迁移前后结果：
   - 词条总数
   - 核心字段完整率
   - 详情页字段渲染完整率
   - 搜索摘要一致性
3. 在验证通过后，再评估清理旧字段和旧映射逻辑的窗口。
4. S3 资源替换与客户端接入切换应分开执行，避免“资源格式切换”和“应用链路切换”同时发生。

## 风险评估
- **风险 1：标准词本通过结构验收，但语义质量仍不均衡**
  - **等级**: 高
  - **说明**: 当前批次可确认结构合规，但个别词条仍可能存在词性、译文、例句自然度等内容质量问题。
  - **缓解措施**: 将“结构验收通过”与“语义质量验收”分开；S3 替换后继续安排抽样质检，不把内容质量问题误判为 schema 问题。

- **风险 2：数据库迁移影响收藏、任务词和学习记录关联**
  - **等级**: 高
  - **说明**: 现有业务链路对 `words` 表和旧字段存在隐式依赖。
  - **缓解措施**: 优先新增列与适配层，迁移前备份数据库，迁移脚本必须支持幂等执行。

- **风险 3：UI 改造扩大到业务逻辑层**
  - **等级**: 中
  - **说明**: 详情页和学习页会天然牵动播放、切词、收藏、任务调度等行为。
  - **缓解措施**: 先拆渲染适配层，再调整交互主流程；严格区分“展示模型变化”和“学习逻辑变化”。

- **风险 4：历史兼容逻辑持续存在，导致新旧模型长期并行**
  - **等级**: 高
  - **说明**: 若没有明确收敛边界，重构后仍可能继续往 legacy 映射中补丁式加字段。
  - **缓解措施**: 规定所有新词本仅接受 `wordbook.v1`；旧格式只允许走离线转换工具，不再直接进入主导入链路。

## 依赖项
- 已完成的 `wordbook.v1` 标准文档与模板
- 已完成基础规整的标准化词本批次
- 待补充的 schema 校验器
- 待补充的迁移 diff 工具
- 待补充的数据库迁移与 UI 回归测试

## 当前阶段建议
建议按以下顺序推进：
1. 先固化标准 DTO、校验器和导入审计输出。
2. 再做数据库职责调整与迁移脚本。
3. 最后分批替换应用模型和 UI 消费链路。

这样可以避免在标准输入尚未完全固化前，就过早大面积改动业务层。
