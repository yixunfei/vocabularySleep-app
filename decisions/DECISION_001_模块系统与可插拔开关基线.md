# ADR-001：模块系统与可插拔开关基线

## 状态
已接受

## 背景
当前项目长期演进后出现以下问题：
- 单体状态与跨域服务耦合过高，功能改动成本持续上升。
- 顶层导航与 toolbox 入口固定写死，不支持模块化启停。
- 阶段性重构缺乏统一的模块抽象与开关持久化规范。

## 决策
1. 引入 `core/module_system`，作为模块治理统一入口。
2. 模块开关状态采用 `SettingsService + settings(module_toggles_v1)` 持久化，默认全启用。
3. 运行时通过 `ModuleRuntimeGuard` 执行访问判定（含父子模块关系）。
4. 首批将开关接入：
   - 底部导航可见性
   - toolbox 子工具入口可见性
   - 设置中心模块管理页
5. 启动链路先接入 `ProviderScope`，为后续 Riverpod 迁移建立容器基线。

## 备选方案
1. 继续沿用硬编码入口与散落布尔配置
   - 优点: 改动最小
   - 缺点: 无法支撑后续可插拔与模块独立目标
2. 直接引入远程配置平台做模块开关
   - 优点: 控制粒度更高
   - 缺点: 引入外部依赖与复杂度，不适合作为本轮基线

## 后果
- 正面:
  - 模块启停语义统一，导航/工具入口可按配置收敛。
  - 为仓库分层与状态迁移提供稳定的模块边界。
- 负面:
  - 旧测试桩需要同步新接口。
  - 当前仍处于“入口层可插拔”阶段，服务层懒初始化需要后续继续推进。
