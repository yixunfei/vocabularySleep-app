## 核心原则

### 优先级排序
1. **移动端优先**: 手机屏幕空间不足时，优先使用抽屉（Drawer）折叠和子页面。
2. **突出主题**: 保留并突出最重要的主题内容。
3. **稳健性**: 代码必须健壮，模块必须可维护。
4. **文档先行**: 修改前先创建计划，修改时对照计划，完成后先更新 changelog，再回看当前进度。
5. **模块化低耦合**: 对于综合性应用尽可能降低各模块间的耦合程度，非必要不进行强侵入式编码。
6. **规范可执行**: 设计规范和结构规范必须能直接指导改动，而不是仅作为参考摆设。

### 风险意识
- 识别潜在风险并文档化。
- 评估每项决策的利弊。
- 对歧义代码添加必要注释。
- 当视觉改动可能牵涉逻辑边界时，先收口设计判断，再落代码。

## 移动端体验规范

### 屏幕空间管理
- 主题舞台信息尽可能在一屏内展示，必要时可折叠或拆入子级页面。
- 必要时进行页面子级拆分，避免超长页面堆叠。
- 注重移动端用户操作习惯与单手操作热区。
- 尽可能避免水平滑动；若内容超出屏幕，优先换行、缩写展开或抽屉展示。
- 注意避免层级关系和手势冲突，例如主舞台内部操作与页面纵向滚动冲突。

### 响应式设计
- 基准设计宽度: `375dp`（iPhone SE）。
- 最大内容宽度: `600dp`。
- 关键操作按钮: 最小触摸区域 `48dp`。

### 首屏信息要求
- 用户进入页面后，应在首屏快速识别“当前工具是什么、当前状态是什么、下一步能做什么”。
- 主舞台顶部信息区优先于装饰背景优化。
- 加载、错误、选中、禁用等状态必须独立可读，不与正文混排。

## toolbox 设计规范执行要求

### 规范来源
凡涉及 `toolbox` 模块的 UI 精修、样式重构、组件收敛或设计反馈闭环，必须同步遵守以下文档：

- `docs/toolbox_design/TOOLBOX_DESIGN_REVIEW.md`
- `docs/toolbox_design/TOOLBOX_ANIMATION_SPEC.md`
- `docs/toolbox_design/TOOLBOX_UI_STYLE_GUIDE.md`

其中：

- `TOOLBOX_DESIGN_REVIEW.md` 用于理解问题来源与优先级。
- `TOOLBOX_ANIMATION_SPEC.md` 用于约束动效节奏、缓动与性能边界。
- `TOOLBOX_UI_STYLE_GUIDE.md` 是当前生效的视觉基线，后续 toolbox UI 改动默认以它为准。

### 执行边界
当任务明确要求“只动 UI、不动逻辑”时，仅允许：

- Widget 树结构调整。
- 展示层文件拆分与组织优化。
- 样式常量、设计 token、共享组件抽取。
- 页面信息层次、卡片语言、动效参数、间距与对齐优化。
- 不改变语义的展示文案容器重排。

在该前提下禁止：

- 修改播放、计时、持久化、状态机和数据模型逻辑。
- 改变事件触发语义、路由行为和业务判断分支。
- 借“样式重构”之名顺手改动状态来源或逻辑入口。

### toolbox 页面统一要求
- 页面骨架优先遵循“导航区 → 主舞台信息区 → 状态/模式区 → 主操作区 → 辅助设置区 → 次级反馈区”。
- 主舞台信息区必须具备稳定承托层，不得直接把关键文字悬浮在复杂背景上。
- 同类控件在不同 toolbox 页面中应优先复用同一套卡片、pill、chip、按钮和面板语言。
- 氛围层要柔和、缓慢、克制，避免高频闪烁、机械重复和过度对称。
- 新设计若会影响多页共用样式，应优先抽取共享层，而不是继续复制粘贴。

### 视觉验收底线
每次 toolbox UI 迭代完成后，至少确认：

- 首屏信息是否清晰。
- 主操作是否可触达、可识别。
- 选中态/错误态/加载态是否独立可读。
- 卡片、边框、阴影、圆角是否与已收口页面一致。
- 是否严格遵守“只动 UI、不动逻辑”的边界。

## 编码与本地化

### 编码规范
```yaml
文件编码: UTF-8（强制）
换行符: LF（Unix 风格）
缩进: 2 空格
命名: snake_case（变量/函数）, PascalCase（类/组件）
```

### 国际化（i18n）
所有用户可见文本必须使用统一的国际化管理，拒绝硬编码用户可见字符串。

## 文档管理

### 文档目录结构
```text
├── AGENTS.md
├── docs/
├── PROJECT_DOMAIN.md
├── plans/
│   ├── PLAN_001_xxx.md
│   └── PLAN_002_xxx.md
├── records/
│   ├── record_001_xxx.md
├── changelogs/
│   └── CHANGELOG.md
├── modules/
│   ├── module_01/
│   ├── module_02/
│   └── module_03/
└── decisions/
    └── DECISION_001_xxx.md
```

### 文档更新规则
1. `PROJECT_DOMAIN.md`: 任何项目范围修改时必须同步检查。
2. 引用链路检查: 修改后验证相关文档的引用完整性。
3. 版本同步: 文档版本号与代码版本保持一致。
4. 设计规范同步: 若 `toolbox` 视觉基线发生变化，必须同步检查 `docs/toolbox_design/` 与 `AGENTS.md` 是否仍一致。
5. 规范落地优先: 若文档规范已成为执行基线，不允许只更新代码而不更新文档。

## 代码规范

### 模块拆分原则
| 模块 | 职责 | 文件大小建议 |
|------|------|-------------|
| 页面（Pages） | 页面布局和状态编排 | < 200 行 |
| 组件（Widgets） | 可复用 UI 组件 | < 100 行 |
| 服务（Services） | 业务逻辑 | < 300 行 |
| 模型（Models） | 数据结构 | < 150 行 |
| 仓库（Repositories） | 数据访问 | < 200 行 |

### 禁止事项
- ❌ 单文件超过 500 行。
- ❌ 大类 / 上帝类。
- ❌ 未分类的通用工具类。
- ❌ 用户可见文本硬编码。
- ❌ 在 UI 收敛任务中混入逻辑改动。

### 注释规范
```dart
// ============ 风险标注示例 ============
// [风险] 选择此实现的原因：
// 1. 当前业务场景需要...
// 2. 替代方案 X 存在...问题
// 3. 如未来需求变更，需考虑...
// ======================================

// [歧义] 这里的命名解释：
// "xxx" 指的是...，而非...（因为...）

// [TODO] 后续优化点：
// 当...时，可考虑使用...替代
```

### 命名规范
```dart
// 页面: xxx_page.dart
// 组件: xxx_widget.dart / xxx_tile.dart / xxx_button.dart
// 模型: xxx_model.dart
// 枚举: xxx_type.dart / xxx_status.dart
// 常量: xxx_const.dart
```

### UI 结构建议
- 页面文件负责结构编排，不负责堆积大量局部样式常量。
- 复杂页面优先拆成 `header / section / panel / controls / tokens / copy` 等子文件。
- 相同视觉语义优先使用共享组件，不重复造型。

## 问题诊断

### 问题修复流程
```text
问题报告
    ↓
追根溯源
    ↓
可能原因列表
    ↓
原因分析
    ↓
根本原因
    ↓
修复方案
    ↓
验证测试
    ↓
更新 Changelog
```

### 日志规范
```dart
LogService.d('模块名', '操作描述', data: {...});
LogService.e('模块名', '错误描述', error: e, stackTrace: st);
```

## 资源优化

### 包体大小控制
| 资源类型 | 策略 | 配置方式 |
|----------|------|----------|
| 图片 | WebP 格式，按需加载 | `flutter: assets:` 分组 |
| 字体 | 按需加载，subset | `assets/fonts/` 分离 |
| 视频/音频 | 远程 CDN | `cached_network_image` |
| 大文件 | 按需加载 + 本地缓存 | `dio` |

### 推荐开源库
当当前方案存在更成熟、更低维护成本的开源方案时，应主动提出参考建议。

## 决策流程

### 模糊概念处理
```text
用户提出模糊需求
    ↓
清晰条例化梳理
    ↓
评估反馈
    ├── 可能风险？
    ├── 更好方案？
    └── 约束条件？
    ↓
形成文档
    ↓
用户确认
    ↓
制定方案
    ↓
编码实现
```

### 技术决策记录（ADR）
```markdown
# ADR-001: {决策标题}

## 状态
提议中 / 已接受 / 已废弃

## 背景
{问题描述}

## 决策
{最终方案}

## 备选方案
1. 方案 A: ...
2. 方案 B: ...

## 后果
- 正面: ...
- 负面: ...
```
